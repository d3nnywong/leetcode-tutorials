这是一道经典的「递归入门题」——如果你完全没编程基础也别担心，「递归」这个词我们等下就会用大白话讲清楚。 这题难的不是代码量（其实只有几行），而是要敢于「相信」递归： 只要管好「当前这一个节点」该干什么，剩下的事情全部放心交给递归自己去完成。

🧩 第一步：先搞懂什么是「二叉树」，「翻转」又是个什么操作

如果你没接触过「树」这种结构，可以把它想象成一张家谱图：最上面是老祖宗， 往下一代代分叉，每个人下面可能挂着自己的孩子。二叉树就是这类家谱图里最简单的一种—— 规定每个人（我们管它叫「节点」）最多只能有两个孩子，习惯上分别叫「左孩子」和「右孩子」。 最顶上那个没有父母的节点叫根节点（root）；没有任何孩子的节点，叫叶子节点—— 就像家谱画到底、没有后代的那一代人。

搞懂了二叉树，「翻转」就好理解了：把它变成镜像对称的样子——就像照镜子一样， 每一个节点的左孩子和右孩子互换位置，而且不只是根节点，每一层、每一个节点 都要这么换。

root[4,2,7,1,3,6,9]

这串数字是按「一层层、从左到右」的顺序写出这棵二叉树的：根节点 4 的左孩子 2、右孩子 7 互换； 2 的左孩子 1、右孩子 3 互换；7 的左孩子 6、右孩子 9 互换……整棵树左右镜像之后变成 [4,7,2,9,6,3,1]。

🔑 第二步：把大问题拆成「一样的小问题」

直接去想「整棵树怎么镜像」会很绕——树可能有好几层，光凭眼睛看规律很容易晕。 这里要请出解决「树」这类问题的头号武器：递归。

所谓递归，就是让一个函数在执行的过程中调用它自己，去解决一个 「规模更小、但本质上一模一样」的子问题——有点像两面镜子对照，镜子里还是一面镜子； 又有点像拆俄罗斯套娃：打开最外面那个，里面还是一个长得一模一样、只是更小的娃娃， 一层层拆下去，直到拆到最里面那个拆不动的娃娃为止。

用在这道题上：「翻转一整棵二叉树」和「翻转它左边（或右边）那一小坨子树」其实是同一件事， 只是规模变小了。所以我们完全不需要一次想清楚全部层级，翻转一棵树可以拆成三句话：

  • 树是空的？什么都不用做，直接返回——这是递归的「终止条件」， 也就是俄罗斯套娃里最里面那个拆不动的娃娃。
  • 不是空的？先把当前节点的左右孩子换一下位置。
  • 换完之后，新的左子树和新的右子树各自也要被翻转—— 而「翻转一棵子树」和「翻转整棵树」是同一件事，所以直接对它们调用 同一个函数就行，这就是「递归调用自己」。
不用一次想清楚全部层级：你只需要相信——只要每个节点都老老实实 「交换左右孩子、再把同样的事丢给两个孩子去做」，递归会自动把这件事铺满整棵树， 一路传到最深的叶子节点为止。

⚡ 第三步：看动画——前序递归，一边交换一边往下走

「遍历」这个词，说白了就是按某种顺序把树上所有节点都访问一遍。 动画演示的是前序遍历的顺序：规则是「先处理当前节点自己，再去左边，再去右边」 （可以记成口诀「中 - 左 - 右」）。放到这道题里就是：进入一个节点 → 交换它的左右孩子 → 再分别递归进入「新的左子树」「新的右子树」。 注意连线两端的节点其实从未改变，变的只是某个节点把谁当成左孩子、谁当成右孩子—— 所以你会看到两棵子树在交换的瞬间，左右换了个位置。

正在访问的节点 正在交换左右孩子 叶子节点（无需交换） 全部翻转完成

试试把输入改成 2,1,3，或者输入 [] 看看空树的特殊情况。

☕ 第四步：写成 Java 代码

class Solution {
    public TreeNode invertTree(TreeNode root) {
        // 终止条件：空节点直接返回（递归到叶子的下一层时触发）
        if (root == null) {
            return null;
        }

        // 单层逻辑：交换当前节点的左右孩子
        TreeNode tmp = root.left;
        root.left = root.right;
        root.right = tmp;

        // 子问题：对换完之后的新左子树、新右子树做同样的事
        invertTree(root.left);
        invertTree(root.right);

        return root;
    }
}
时间复杂度 O(n)

每个节点恰好被访问一次，做一次常数时间的交换。

空间复杂度 O(h)

h 是树的高度，这部分空间来自递归调用栈——你可以把「栈」想象成一摞盘子， 只能从最上面摞新盘子，也只能从最上面拿走盘子（后进先出）。每次函数调用自己， 就像往这摞盘子上再摞一个「还没执行完、等会儿要回来接着做」的盘子；树有多深， 这摞盘子最多就摞多高。最坏情况（树退化成一条链）是 O(n)，比较均衡的树是 O(log n)。

🚀 小结

套路：看到「对树的每个节点都做同一件事」，先想递归三件套—— 终止条件、当前节点要做的事、把剩下的丢给子问题。这题的子问题恰好就是「翻转一棵（更小的）树」， 和原问题一模一样，递归用起来特别顺。 如果暂时不想用递归，也可以换一种思路——BFS（广度优先搜索）：借助一个 队列（想象成排队买票，先进去的先被服务）把节点一个个取出来处理； 或者自己维护一个显式的栈（一摞盘子，后进先出）来代替递归本来会用到的调用栈。 两种思路的本质和递归完全一样，只是把「递归自动帮你记住还没处理完的节点」这件事， 换成了自己动手拿一个队列或栈去记。
