0 到 n 一共 n+1 个数，数组里只装了 n 个、独一无二、丢了一个。 不用排序、不用哈希表，一个加法公式就能把丢的那个数「算」出来。

🧩 第一步：题目到底要什么

给你一个长度为 n 的数组 nums，里面装着 [0, n] 这个范围内的 n 个互不相同的数。也就是说， [0, n] 里一共有 n + 1 个数（0、1、2 … 一直到 n）， 但数组只装得下 n 个，所以正好少了一个。 你的任务就是把这个丢失的数字找出来。

nums[3, 0, 1]

n = 3，范围是 [0, 3]，里面应该有 0、1、2、3 共 4 个数。 数组里出现了 3、0、1，唯独没有 2 → 答案是 2。

🐢 第二步：能想到的几种朴素做法

最直接的想法：先把数组排序（也就是把这堆数从小到大重新排好队）， 排好之后 0、1、2、3… 应该是紧挨着的，从头开始逐个比对， 哪个位置对不上「应该出现的数」，那个数就是丢的那个。 但排序本身要花时间，数越多排序越慢（记作 O(n log n)， 你可以先简单理解成「比线性扫一遍稍微慢一点点」）。

另一种办法：用一个哈希表（或者更简单的布尔数组）—— 你可以把它想象成一本「查号簿」，见到一个数字就在簿子里把它「打勾」， 最后再翻一遍 0..n，谁没被打勾，谁就是丢的那个。 这样只需要扫两遍数组（记作 O(n) 时间，「n」就是数组的长度， 意思是耗时跟数组长度成正比），但代价是要额外准备一本「查号簿」来记录， 也就是需要额外的存储空间（记作 O(n) 空间）。

能不能更省：这两种办法都能做对，但题目的进阶要求是—— 时间只要 O(n)（扫一遍就行）、额外空间只要 O(1) （不许排序，也不许再准备一本「查号簿」，只能用几个变量）。 我们需要一个不用「记录每个数字见没见过」、只靠「算」就能得到答案的办法。

⚡ 第三步：用「高斯求和」一步算出答案

先讲个小故事帮你理解这一步：相传数学家高斯小时候，老师让全班把 1 加到 100，小高斯很快就答出来了——他发现把 1 和 100 配一对、2 和 99 配一对……每一对加起来都是 101，一共 50 对，所以答案是 101 × 50，根本不用一个个去加。这就是「配对求和」的小技巧， 任何一串连续整数的和，都能用类似的公式秒算出来。

回到我们的题：关键观察是——如果数组没有丢任何数， 它就应该正好是 0, 1, 2, …, n 这连续的 n + 1 个数 （题目只丢了一个，去掉它就剩下 n 个）。 而 0 到 n 这一串连续整数「应该有」的总和， 正好可以用高斯的配对公式直接算出来：

公式expectedSum = n × (n + 1) / 2

这是「应该有」的总和。再把数组里实际出现的数一个个加起来，得到 actualSum。因为只丢了一个数，「应该有」的总和一定比「实际」的总和 多出丢掉的那个数——多出来的那部分差值，就是我们要找的答案： missing = expectedSum − actualSum。

代入第一步的例子看一眼：nums = [3, 0, 1]，n = 3。 「应该有」的总和 expectedSum = 3 × 4 / 2 = 6 （对应 0+1+2+3）；数组里「实际」的总和 actualSum = 3 + 0 + 1 = 4。 两者一减：6 − 4 = 2，正好就是丢掉的 2，一步到位，完全不用排序或记表。
当前正在累加的 nums[i] 已经加进 actualSum 最终算出的丢失数字

试试把数组改成 0,1（答案应该是 2）或更长的乱序数组， 看 actualSum 怎么一步步逼近 expectedSum 少的那一块。

☕ 第四步：写成 Java 代码

class Solution {
    public int missingNumber(int[] nums) {
        int n = nums.length;
        int expectedSum = n * (n + 1) / 2; // 0..n 这 n+1 个数「应有」的和
        int actualSum = 0;
        for (int num : nums) {
            actualSum += num; // 数组里「实际」出现的数之和
        }
        return expectedSum - actualSum; // 差值就是丢失的那个数
    }
}
时间复杂度 O(n)

数组只扫一遍把数加起来，高斯公式那部分是 O(1)。

空间复杂度 O(1)

全程只用了 expectedSum / actualSum 两个变量，不需要排序或额外容器。

🚀 小结

套路：当题目告诉你「一组数本该是连续/完整的某个集合，但缺了一个」时， 先想想这个「本该有」的整体有没有现成的公式（求和、计数都行）， 再用「应有 − 实际」这种差值思路一步到位，往往比排序或额外建表更省时间和空间。
拓展一下：这道题还常用按位异或来做。 计算机里每个整数都是由一串 0 和 1 组成的（叫「二进制」）， 「异或」是一种逐位比较的运算：两个数同一位上的 0/1 一样就得 0，不一样就得 1。 它有个很好用的性质——同一个数异或它自己，结果一定是 0 （比如 5 ^ 5 = 0），而任何数异或 0 还是它本身。 所以只要把 0..n 和数组里的每个数全部异或一遍， 成对出现的数字会两两抵消变成 0，最后剩下的那个「单身」的数字，就是丢失的答案。 换个工具，本质还是同一个套路：找一个「全部合并后能自动消掉重复部分」的运算。
