0 到 n 一共 n+1 个数，数组里只装了 n 个、独一无二、丢了一个。 不用排序、不用哈希表，一个加法公式就能把丢的那个数「算」出来。

🧩 第一步：题目到底要什么

给你一个长度为 n 的数组 nums，里面装着 [0, n] 这个范围内的 n 个互不相同的数。也就是说， [0, n] 里一共有 n + 1 个数（0、1、2 … 一直到 n）， 但数组只装得下 n 个，所以正好少了一个。 你的任务就是把这个丢失的数字找出来。

nums[3, 0, 1]

n = 3，范围是 [0, 3]，里面应该有 0、1、2、3 共 4 个数。 数组里出现了 3、0、1，唯独没有 2 → 答案是 2。

🐢 第二步：能想到的几种朴素做法

最直接的想法：把数组排序，然后从 0 开始逐个比对， 哪个位置对不上就是丢的那个（O(n log n)）； 或者用一个哈希表 / 布尔数组记下出现过的数字， 再扫一遍 0..n 看谁没出现（O(n) 时间，但要 O(n) 额外空间）。

能不能更省：这两种办法都能做对，但题目的进阶要求是—— 时间 O(n)、空间只用 O(1)（不许排序，也不许额外开数组/哈希表）。 我们需要一个不用「记录」、只靠「算」就能得到答案的办法。

⚡ 第三步：用「高斯求和」一步算出答案

关键观察：如果数组没有丢任何数，它就应该正好是 0, 1, 2, …, n 这 n + 1 个数（去掉某一个就剩 n 个）。 而 0 到 n 这一串连续整数的和，可以用大数学家高斯小时候的公式 直接算出来，不用一个个加：

公式expectedSum = n × (n + 1) / 2

这是「应该有」的总和。再把数组里实际出现的数加起来，得到 actualSum。因为只丢了一个数，多算的那部分差值， 正好就是丢失的那个数：missing = expectedSum − actualSum。

当前正在累加的 nums[i] 已经加进 actualSum 最终算出的丢失数字

试试把数组改成 0,1（答案应该是 2）或更长的乱序数组， 看 actualSum 怎么一步步逼近 expectedSum 少的那一块。

☕ 第四步：写成 Java 代码

class Solution {
    public int missingNumber(int[] nums) {
        int n = nums.length;
        int expectedSum = n * (n + 1) / 2; // 0..n 这 n+1 个数「应有」的和
        int actualSum = 0;
        for (int num : nums) {
            actualSum += num; // 数组里「实际」出现的数之和
        }
        return expectedSum - actualSum; // 差值就是丢失的那个数
    }
}
时间复杂度 O(n)

数组只扫一遍把数加起来，高斯公式那部分是 O(1)。

空间复杂度 O(1)

全程只用了 expectedSum / actualSum 两个变量，不需要排序或额外容器。

🚀 小结

套路：当题目告诉你「一组数本该是连续/完整的某个集合，但缺了一个」时， 先想想这个「本该有」的整体有没有现成的公式（求和、异或、计数都行）， 再用「应有 − 实际」这种差值思路一步到位，往往比排序或额外建表更省时间和空间。 这道题也常用按位异或来做（把 0..n 和数组元素全部异或一遍， 相同的数两两抵消，剩下的就是答案），思路是同一个套路的另一种实现。
