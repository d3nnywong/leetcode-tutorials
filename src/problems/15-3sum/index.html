两数之和的「升级版」：从找两个数变成找三个数。核心套路是先把数组排序， 再把其中一个数当「锚点」，剩下的就退化成你已经会的双指针找两数之和。

🧩 第一步：题目到底要什么

给你一个整数数组 nums，要你找出所有「三个数加起来等于 0」 的组合（不是下标，是数值组成的三元组），而且不能有重复的组合。

nums[-1, 0, 1, 2, -1, -4]

-1 + 0 + 1 = 0，-1 + (-1) + 2 = 0 → 答案是 [[-1,-1,2],[-1,0,1]]， 顺序不重要，但每一组三元组内部和组与组之间都不能重复。

🐢 第二步：最直接的笨办法

最容易想到的：写三层循环，把数组里所有 i, j, k 三个下标能组成的组合都试一遍， 看哪三个数加起来正好是 0。这样容易凑出重复的三元组（比如同样的三个数字， 从不同下标出发会被数好几次），所以还得拿一个「集合」——可以理解成一个不允许放进 重复东西的口袋，一样的组合放第二次会被自动挡在外面——把结果去重。 这一套是 O(n³)，数组长度能到 3000，三次方就是 270 亿次，电脑也要跑好一阵子。

慢在哪：每确定一个 i，后面两个数还要再来一次「暴力配对」—— 一个个组合都试一遍。但仔细想想，「从剩下的数里找两个数，让它们加起来等于某个目标值」， 这不就是两数之和吗？两数之和已经有更快的办法了，这里应该也能借过来用。

⚡ 第三步：排序后，退化成「双指针版两数之和」

先把数组从小到大排序。排序能带来两个好处：一是相同的数字会挨在一起， 待会儿去重方便很多；二是数字有了「从小到大」的顺序之后，就能用上一种叫 双指针的小技巧——想象在数组的最左边和最右边各伸出一根手指头 （代码里用两个变量 l（left，左指针）和 r（right，右指针）记住它们指着哪个位置）， 然后看两根手指头指的数加起来是偏大还是偏小，让其中一根手指往中间挪一步， 这样一步步把「可能的范围」夹小，直到两根手指碰到一起。这比把每一对都死板地试一遍要快得多。

具体用到这题上：先固定第一个数 nums[i] 当成一个不动的「参照点」 （我们叫它「锚点」，就是先把它钉住，剩下的事情都围着它转）。这样问题就变成了： 在 i 右边的这一段区间里，找两个数 nums[l] + nums[r] = -nums[i]—— 这不就是两数之和吗？只不过这次不用哈希表，改用双指针来找：

  • 三数之和 等于 0 → 说明这一组就是答案，记下来；然后左右指针各跳过和刚才相同的值再继续夹逼（这一步就是在去重，避免把一模一样的三元组记录两次）。
  • 三数之和 小于 0 → 说明和太小了，得换一个更大的数，让左指针往右挪一步。
  • 三数之和 大于 0 → 说明和太大了，得换一个更小的数，让右指针往左挪一步。

另外还有两个「剪枝」的小技巧——剪枝的意思是：提前判断出某条路根本不可能有答案， 就干脆不去试，像剪掉一根注定长不出果子的树枝，省下大量白费的功夫。这里用到两处： 排序后如果 nums[i] > 0，说明它和它右边的数（都比它大或相等）加起来只会更大， 不可能凑出 0，可以直接结束整个循环；如果 nums[i] 和上一个锚点的值相同， 就跳过它——因为上一轮已经拿这个数当过锚点试过一遍了，再试一次只会算出重复的三元组。

锚点 nums[i] 左右指针 l / r 命中的三元组 跳过重复 / 提前结束

试试把数组改成 0,0,0 或 0,1,1，看它怎么处理「正好一组」和「无解」。

☕ 第四步：写成 Java 代码

import java.util.*;

class Solution {
    public List<List<Integer>> threeSum(int[] nums) {
        Arrays.sort(nums); // 排序后才能用双指针、方便跳过重复值
        List<List<Integer>> ans = new ArrayList<>();
        int n = nums.length;
        for (int i = 0; i < n - 2; i++) {
            if (nums[i] > 0) break; // 最小的数都大于 0，后面只会更大
            if (i > 0 && nums[i] == nums[i - 1]) continue; // 跳过重复锚点

            int l = i + 1, r = n - 1;
            while (l < r) {
                int sum = nums[i] + nums[l] + nums[r];
                if (sum == 0) {
                    ans.add(Arrays.asList(nums[i], nums[l], nums[r]));
                    l++; r--;
                    while (l < r && nums[l] == nums[l - 1]) l++; // 跳过重复左值
                    while (l < r && nums[r] == nums[r + 1]) r--; // 跳过重复右值
                } else if (sum < 0) {
                    l++; // 太小，左指针右移找更大的数
                } else {
                    r--; // 太大，右指针左移找更小的数
                }
            }
        }
        return ans;
    }
}
时间复杂度 O(n²)

排序 O(n log n)；外层枚举锚点 O(n)，内层双指针再扫一遍 O(n)，合计 O(n²)。

空间复杂度 O(log n) ~ O(n)

不计返回结果的话，主要是排序算法自己内部用到的一点点临时空间 （排序内部会「自己调用自己」去处理更小的一段数组，系统需要记一笔 「调用到哪一步了」的小账本，这部分占用的空间大致是 O(log n) ~ O(n)， 可以先不必深究细节）；返回的三元组列表本身有多大，习惯上不计入复杂度。

🚀 小结

套路：遇到「找 k 个数满足某种和」的题，常见思路是先排序， 再固定前 k-2 个数，把问题降维成「双指针找两数之和」。 三数之和把两数之和从 O(n) 升级成了「固定一维 + 双指针扫一维」的 O(n²)， 四数之和、最接近的三数之和都是同一个套路的变体。
