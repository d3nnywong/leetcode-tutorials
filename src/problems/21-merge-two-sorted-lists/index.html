两条已经排好序的「队伍」，怎么合并成一条更长的有序队伍？ 看完你会明白：两个指针 + 一个哨兵节点，就能一遍扫描拼好。

🧩 第一步：题目到底要什么

先弄清楚题目里的「链表」是什么东西。数组大家可能更熟悉：一串数字紧挨着排在一起， 想拿第 5 个直接跳过去就行。链表不是这样存的——它更像一串糖葫芦， 一颗一颗的山楂（我们管它叫「节点」）用竹签串起来，但每颗节点其实是分开摆放的， 它自己只记两件事：自己的值（val），和 下一颗节点在哪儿（next，可以理解成「牵着下一颗的手」）。 想找第 5 颗节点，没法直接跳过去，只能从第 1 颗开始，顺着 next 一颗一颗数过去。 串到最后一颗，它没有下一颗可牵了，next 就是 null （意思是「没有下一个了，到头了」）。

弄清链表长什么样，题目就好懂了：给你两个已经从小到大排好序的链表 list1 和 list2，要把它们合并成 一个新的、依然从小到大排序的链表。而且要求是「拼」出来的—— 不是把值抄一遍去建一堆新节点，而是直接把原来两条链表上现成的节点重新接一下线 （改一改它们的 next 牵向谁），拼成一条新链表。

list1[1, 2, 4]
list2[1, 3, 4]

就像两摞已经从小到大排好的扑克牌，每次都从两摞牌顶各看一眼，拿走更小的那张， 最后拼出一摞完整有序的牌：[1, 1, 2, 3, 4, 4]。

🐢 第二步：最直接的笨办法

最容易想到的：把两个链表从头到尾都走一遍，把沿途每颗节点的值都倒进一个数组， 再用 sort() 排个序，最后照着排好的值重新建一条新链表（一颗一颗新建节点， 把 next 依次串起来）。这样确实能做出正确答案，但既浪费了「两个链表本来 就各自有序」这个白送的条件，又多花了一次排序的功夫，还得凭空造出一堆新节点。

慢在哪：收集 + 排序是 O((m+n) log(m+n))。 两个链表本身已经各自有序，真正需要的只是把它们「交错合并」起来，根本不需要重新排序。

⚡ 第三步：双指针 + 哨兵节点，一遍扫描拼好

两个链表各自已经有序，那我们完全不用「收集再排序」这么绕。想象两个人分别站在 list1、list2 队伍的最前面，我们叫他们 p1、 p2——这俩其实就是两个普通变量，各自「记着」自己现在站在哪颗节点上 （所以叫「指针」，你可以理解成两个书签，分别夹在两条链表里还没被接走的位置）。 每一步只比较 p1、p2 当前这两颗节点的值，谁小就把谁接到 结果链表末尾，再让对应的指针沿着它的 next 往前挪一步。

接的时候有个小麻烦：结果链表的第一颗节点比较特殊——它前面还没有 任何节点，没法说「接在谁后面」。为了不专门为这一颗节点写一堆判断，常用的小技巧是： 先凭空造一个不存真实数据、纯粹占位的哨兵节点 dummy （也叫 dummy head，可以理解成排在队伍最前面、自己不算数、专门用来「起个头」的假人）。 再用一个 tail 指针（同样只是个变量，记着「结果链表现在拼到哪儿了」） 指向结果链表目前的最后一颗节点：每接上一颗新节点，就把 tail 也跟着移到这颗新节点上，永远保持「牵着队尾」。

这样一步步走下去，两个链表里迟早会有一个先走到 null （前面说过，这表示「没有下一个了，到头了」）。这时候另一个链表剩下的部分 根本不用再逐个比较、逐个接——它本身已经是排好序的一整段，直接把 tail.next 指过去，就等于把剩下所有节点一次性全部接完了。

p1 / p2 当前指针 本步选中的较小节点 刚接入结果链表的节点

试试把 list1 改成 1,2,4,8、list2 改成 2,3， 看其中一条先走到 NULL 后，剩下的整段是怎么直接接上去的。

☕ 第四步：写成 Java 代码

// 链表节点定义（LeetCode 已提供，这里展示结构）
class ListNode {
    int val;
    ListNode next;
    ListNode() {}
    ListNode(int val) { this.val = val; }
    ListNode(int val, ListNode next) { this.val = val; this.next = next; }
}

class Solution {
    public ListNode mergeTwoLists(ListNode list1, ListNode list2) {
        ListNode dummy = new ListNode(-1); // 哨兵节点，避免单独处理头节点
        ListNode tail = dummy;             // tail 始终指向结果链表的最后一个节点

        while (list1 != null && list2 != null) {
            if (list1.val <= list2.val) {
                tail.next = list1;
                list1 = list1.next;
            } else {
                tail.next = list2;
                list2 = list2.next;
            }
            tail = tail.next;
        }

        // 谁还有剩余，它本身就是有序的，直接整段接上去
        tail.next = (list1 != null) ? list1 : list2;

        return dummy.next;
    }
}
时间复杂度 O(m + n)

m、n 分别是两个链表的长度，每个节点最多被比较、移动一次。

空间复杂度 O(1)

只是把已有节点的 next 指针重新接线，没有额外开辟节点空间（不计返回结果本身）。

🚀 小结

套路：合并两个「已经有序」的序列时，不需要重新排序——用两个指针各自 沿着链表往前扫、每步取较小的那颗节点接走即可，这就是归并排序里「合并」步骤的原型 （把两支已经排好队的小队伍，合并成一支排好队的大队伍）。再加上一个 哨兵节点，专门解决「第一颗节点没有『上一颗』可以参照着接」这类边界情况， 是链表题里非常常用的技巧。
