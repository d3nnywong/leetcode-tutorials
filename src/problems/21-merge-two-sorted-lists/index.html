两条已经排好序的「队伍」，怎么合并成一条更长的有序队伍？ 看完你会明白：两个指针 + 一个哨兵节点，就能一遍扫描拼好。

🧩 第一步：题目到底要什么

给你两个已经从小到大排好序的链表 list1 和 list2，要把它们合并成一个新的、依然从小到大排序的链表， 并且新链表是用原来两个链表的节点直接「拼」出来的（不新建节点存值，只是改 next 指针）。

list1[1, 2, 4]
list2[1, 3, 4]

就像两摞已经从小到大排好的扑克牌，每次都从两摞牌顶各看一眼，拿走更小的那张， 最后拼出一摞完整有序的牌：[1, 1, 2, 3, 4, 4]。

🐢 第二步：最直接的笨办法

最容易想到的：把两个链表里所有节点的值都倒进一个数组，sort() 排个序，再用排好的值重新建一条新链表。能做出来，但既浪费了「两个链表本来就有序」 这个条件，又多花了一次排序的功夫。

慢在哪：收集 + 排序是 O((m+n) log(m+n))。 两个链表本身已经各自有序，真正需要的只是把它们「交错合并」起来，根本不需要重新排序。

⚡ 第三步：双指针 + 哨兵节点，一遍扫描拼好

既然两个链表各自有序，那就用两个指针 p1、p2 分别指向 list1、list2 当前还没接走的节点，每一步只比较这两个 指针指向的值，把更小的那个节点接到结果链表末尾，再把对应指针后移一步。

为了不用单独判断「第一个节点要接给谁」，通常先建一个不存数据的哨兵节点 dummy（也叫 dummy head），用一个 tail 指针指向结果链表的最后 一个节点，每次新接一个节点就把 tail 跟着移过去。最后哪个链表先走到 NULL，就把另一个链表剩下的整段直接接上去—— 因为它本来就是有序的，不用再逐个比较了。

p1 / p2 当前指针 本步选中的较小节点 刚接入结果链表的节点

试试把 list1 改成 1,2,4,8、list2 改成 2,3， 看其中一条先走到 NULL 后，剩下的整段是怎么直接接上去的。

☕ 第四步：写成 Java 代码

// 链表节点定义（LeetCode 已提供，这里展示结构）
class ListNode {
    int val;
    ListNode next;
    ListNode() {}
    ListNode(int val) { this.val = val; }
    ListNode(int val, ListNode next) { this.val = val; this.next = next; }
}

class Solution {
    public ListNode mergeTwoLists(ListNode list1, ListNode list2) {
        ListNode dummy = new ListNode(-1); // 哨兵节点，避免单独处理头节点
        ListNode tail = dummy;             // tail 始终指向结果链表的最后一个节点

        while (list1 != null && list2 != null) {
            if (list1.val <= list2.val) {
                tail.next = list1;
                list1 = list1.next;
            } else {
                tail.next = list2;
                list2 = list2.next;
            }
            tail = tail.next;
        }

        // 谁还有剩余，它本身就是有序的，直接整段接上去
        tail.next = (list1 != null) ? list1 : list2;

        return dummy.next;
    }
}
时间复杂度 O(m + n)

m、n 分别是两个链表的长度，每个节点最多被比较、移动一次。

空间复杂度 O(1)

只是把已有节点的 next 指针重新接线，没有额外开辟节点空间（不计返回结果本身）。

🚀 小结

套路：合并两个「已经有序」的序列时，不需要重新排序——用双指针各自 扫一遍、每步取较小者即可，这就是归并排序里「合并」步骤的原型。再加上一个 哨兵节点 简化「头节点要单独处理」的边界情况，是链表题里非常常用的技巧。
