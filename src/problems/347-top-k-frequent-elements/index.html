给一堆数字，找出出现次数最多的 K 个。先学会用哈希表「边走边记」， 再看怎么用一个巧妙的「桶排序」把它从 O(n log n) 降到 O(n)。

🧩 第一步：题目到底要什么

给你一个数组 nums，里面的数字会重复出现。再给你一个整数 k，要你找出出现次数最多的 k 个不同数字， 顺序随意。

nums[1, 1, 1, 2, 2, 3]
k2

1 出现 3 次，2 出现 2 次，3 出现 1 次。出现次数前 2 高的是 1 和 2 → 返回 [1, 2]。

🐢 第二步：最直接的办法，以及它的瓶颈

第一反应通常是：先用哈希表把每个数字出现的次数统计出来（值 → 次数）， 再把这些「值-次数」对按次数从大到小排序，取前 k 个的值。

慢在哪：统计本身只要 O(n)，但排序要 O(u log u)（u 是不同数字的个数，最坏 u = n）。 题目「进阶」明确要求做到优于 O(n log n) —— 我们其实不需要把所有数字都排好序，只需要找出排在前 k 的那一小撮， 这就给了「桶排序」可乘之机。

关键观察：一个数字最多能出现 n 次（数组总长度），出现次数的范围是 [1, n]，是一个有限且已知的小区间。次数本身可以当「下标」用—— 这正是桶排序的适用场景：当「排序的键」范围有限时，开一个数组按键直接归位，不用比较， 一次扫描就能按键的大小分好组。

⚡ 第三步：用「桶排序」一遍搞定

做法分三步：

  1. 扫一遍 nums，用哈希表统计每个数字出现的次数。
  2. 开一个大小为 n + 1 的「桶」数组，下标就是出现次数。 每个数字按它的出现次数，被放进对应下标的桶里（同一个桶可能有多个数字， 因为它们出现的次数一样多）。
  3. 从下标最大（出现次数最多）的桶开始，往下标小的桶扫，边扫边把桶里的数字收进答案， 凑够 k 个就停下。

因为桶的数量是 O(n)，每个数字只会进一个桶，整个过程没有任何比较排序， 时间复杂度是 O(n)——比 O(n log n) 更优，正好满足进阶要求。

☕ 第四步：写成 Java 代码

class Solution {
    public int[] topKFrequent(int[] nums, int k) {
        // 1) 统计频率：值 -> 次数
        Map<Integer, Integer> freq = new HashMap<>();
        for (int x : nums) {
            freq.put(x, freq.getOrDefault(x, 0) + 1);
        }

        // 2) 按频率装桶：下标即出现次数，桶里放达到该频率的数字
        int n = nums.length;
        List<Integer>[] buckets = new List[n + 1];
        for (Map.Entry<Integer, Integer> e : freq.entrySet()) {
            int count = e.getValue();
            if (buckets[count] == null) {
                buckets[count] = new ArrayList<>();
            }
            buckets[count].add(e.getKey());
        }

        // 3) 从高频桶往低频桶收集，凑够 k 个就停
        int[] result = new int[k];
        int idx = 0;
        for (int count = n; count >= 1 && idx < k; count--) {
            if (buckets[count] == null) continue;
            for (int val : buckets[count]) {
                result[idx++] = val;
                if (idx == k) break;
            }
        }
        return result;
    }
}
时间复杂度 O(n)

统计频率一遍、装桶一遍、收集一遍，全程没有比较排序。

空间复杂度 O(n)

哈希表和桶数组最坏情况下都要存下接近 n 个元素。

🚀 小结

套路：当你要排序的「键」（这里是出现次数）的取值范围 有限且已知时，不一定要用通用的比较排序，可以考虑「桶排序」—— 把键直接当下标，开桶分组，一次扫描搞定。这道题还告诉我们： 题目只要前 k 个，往往意味着不需要把全部元素都排好序， 能省的功夫尽量省。
