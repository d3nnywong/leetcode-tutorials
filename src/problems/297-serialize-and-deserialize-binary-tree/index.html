一棵二叉树活在内存里，全靠一个个指针互相牵着。怎么把它原样存进一个文件、 或者从网络那头原样收回来？这道「设计」难题的答案，其实只是两句话： 按固定顺序走一遍树，连「没有孩子」也老老实实写下来。

🧩 第一步：「序列化」到底要干什么

二叉树存在内存里的时候，每个节点身上都带着两个「箭头」——left 和 right，分别指向它的左孩子、右孩子具体存在哪儿（专业说法叫指针， 你可以把它想象成一张写着「下一个节点在哪」的小纸条；没有孩子的地方，纸条就是空的）。 节点们就是靠这样一张张纸条互相牵着，才连成一棵树。 序列化（serialize） 就是把这棵「靠纸条牵着的树」压成一个普通字符串， 这样就能写进文件、存进数据库，或者通过网络发给另一台机器 ——字符串里可没有什么「指针」，全是普普通通的文字。 反序列化（deserialize） 则是反过来——拿到这个字符串， 原样重建出结构、节点值都一模一样的那棵树（也就是重新把一张张纸条接好）。

root[1,2,3,null,null,4,5]

根节点 1，左孩子 2（2 没有孩子），右孩子 3，3 又有左孩子 4、右孩子 5。 目标：把这棵树变成一个字符串，再保证从这个字符串能精确地还原出同一棵树 ——不能多一个节点、不能少一个节点，更不能左右孩子认错。

🔑 第二步：光记节点值，为什么不够

最直接的想法：把树「拍平」成一串数字不就行了？「遍历」的意思就是按某种顺序， 把树上每个节点都访问一遍——比如中序遍历，就是「先走到最左边、访问它、再往右挪一格」这样的顺序。 我们可以中序遍历一遍，把每个节点的值依次写下来。 但这里有个致命问题——同一串数字，可能对应好几棵长得完全不同的树。 光看值的序列，根本不知道谁是谁的孩子、谁挂在左边、谁挂在右边。

关键洞察：要让字符串能被唯一还原回原来的树， 光写「有什么节点」不够，还必须写清楚「树的形状」——也就是每个节点的孩子在哪、 哪里没有孩子。做法是：

  • 按前序遍历（根 → 左子树 → 右子树）的顺序访问节点。 这里的「子树」很好理解：随便挑树上一个节点，把它和它下面挂着的所有节点单独拎出来看， 这一小撮就自成一棵更小的树，这就是「子树」。按前序遍历，第一个写下来的值， 永远是「当前这棵（子）树」的根，反序列化时一读就知道从哪里开始建。
  • 每当走到一个空位置（某个节点没有左孩子或右孩子），也明确写一个 null 占位——它告诉反序列化「这里到此为止，不用再往下读」。

有了 null 占位符，前序序列就变成了唯一对应一棵树的「编码」。 反序列化的时候，我们把字符串按逗号切开，变成一个个小片段——每一个这样的小片段就叫一个 token（比如 "1,2,null" 切开后，"1"、 "2"、"null" 各自都是一个 token）。 只要按写入的顺序依次读 token：读到数字就建一个新节点， 然后递归着建它的左右子树——「递归」就是这个「建树」的函数在建左右子树时， 又调用了它自己去做同一件事（只是问题变小了：从「建一整棵树」缩成「建一棵子树」）， 就像照镜子时镜子里还套着一面镜子，一层一层往下照，直到读到 null 就直接返回、 不再往下建——读的顺序和写的顺序天然对得上，一气呵成。

⚡ 第三步：看动画——先「前序写」，再「前序读」

左边是原始的树：动画按前序顺序逐个访问真实节点（写入它的值）和空位置（写入 null），下方的「纸带」会同步把字符串一格一格填出来。 写完之后，右边的空白区域开始按完全相同的顺序读这条纸带， 每读到一个数字就新建一个节点、挂回它原来的位置，直到整棵树重新长出来。

☕ 第四步：写成 Java 代码

序列化就是前文说的前序遍历：每到一个节点写它的值，每到空位置写 "null"， 用逗号隔开。反序列化时，先把整个字符串按逗号切开，得到一串 token， 再依次放进一个队列里——队列就像排队买票，只能从队尾进、从队头出， 先进去的先被取出来。因为读 token 的顺序和当初写它们的顺序完全一致， 每次从队头取一个 token，就正好对应上前序遍历接下来要访问的那个位置， 读到数字就建节点、递归建它的左右子树，读到 "null" 就返回空。 

/**
 * Definition for a binary tree node.
 * public class TreeNode {
 *     int val;
 *     TreeNode left;
 *     TreeNode right;
 *     TreeNode(int x) { val = x; }
 * }
 */
public class Codec {

    // 序列化：前序遍历，每到一个节点写它的值，每到空位置写 "null"
    public String serialize(TreeNode root) {
        StringBuilder sb = new StringBuilder();
        buildString(root, sb);
        return sb.toString();
    }

    private void buildString(TreeNode node, StringBuilder sb) {
        if (node == null) {
            sb.append("null,"); // 空位置也要老老实实记下来
            return;
        }
        sb.append(node.val).append(",");
        buildString(node.left, sb);  // 先写左子树
        buildString(node.right, sb); // 再写右子树
    }

    // 反序列化：按写入时同样的前序顺序，从一个队列里依次取 token 重建
    public TreeNode deserialize(String data) {
        Deque<String> queue = new ArrayDeque<>(Arrays.asList(data.split(",")));
        return buildTree(queue);
    }

    private TreeNode buildTree(Deque<String> queue) {
        String val = queue.poll(); // 队首永远是「下一个该读」的 token
        if ("null".equals(val)) {
            return null; // 读到占位符，这里没有节点
        }
        TreeNode node = new TreeNode(Integer.parseInt(val));
        node.left = buildTree(queue);  // 接着读的就是它的左子树
        node.right = buildTree(queue); // 左子树读完之后，紧接着就是右子树
        return node;
    }
}
时间复杂度 O(n)

序列化、反序列化都只把每个真实节点（和每个空位置）访问一次。

空间复杂度 O(n)

序列化字符串本身长度是 O(n)；反序列化的队列也存了 O(n) 个 token； 另外递归每往下建一层子树，就会多占用一层「函数调用」的空间——这些占用的空间 叫调用栈，最深能到多少层就决定了它有多大：最坏是 O(n) （树退化成一条链的时候），平衡树则是 O(log n)。

🚀 小结

套路：这是一道「设计」题，没有唯一的标准答案——只要序列化和反序列化 能配成一对、互相还原就行。前序遍历 + null 占位符是最直观的一种：第一个 token 永远是 「当前（子）树的根」，天然适合递归重建。你也完全可以换成层序遍历（BFS， 一层一层地从上往下、从左往右扫），用一个队列装着「待处理」的节点， 代替递归时那个自动帮你记位置的调用栈，思路是一样的：写的时候记下每个节点的孩子（包括 null）， 读的时候按同样的顺序把节点和它们的孩子一一对上。核心只有一句话： 只要写入和读取的顺序、规则严格对称，结构信息就不会丢。
