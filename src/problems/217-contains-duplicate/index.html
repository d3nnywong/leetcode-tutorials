判断一个数组里有没有重复数字，是「用哈希表记住见过什么」这个思路最朴素的起点。 看完你会明白：一个哈希集合，怎么把逐个比较的笨办法压成一遍扫描。

🧩 第一步：题目到底要什么

给你一个整数数组 nums，问里面有没有出现过至少两次的数字。 有就返回 true，所有数字都互不相同就返回 false。

nums[1, 2, 3, 1]

数字 1 出现在下标 0 和下标 3 这两个位置（下标就是数字在数组里 排第几个，从 0 开始数——第 1 个数字下标是 0，第 2 个是 1，以此类推）。 它一共出现了两次 → 返回 true。

🐢 第二步：两种「还不够好」的办法

最直接的：拿每个数去和后面的每个数比较，看有没有相等的 （用编程的话说，就是套两层循环——循环就是「重复做同一件事」， 外层选一个数，内层挨个跟它后面的数比一遍）。这样能做出来，但代价是 O(n²)——意思是数组里有 n 个数，大致要比较 n × n 次。 题目里数组最多能到 10⁵ 那么长，n² 就是上百亿次比较， 电脑也扛不住。

聪明一点：先把数组排序（也就是从小到大重新排一遍）， 相同的数字排序后一定挨在一起，于是只要从左到右扫一遍， 看相邻两个数是否相等就行，O(n log n)。 比暴力好不少，但排序本身就要花 O(n log n)，还会打乱数组原来的顺序。

能不能更快？两层循环慢在「每次都要重新找」；排序快是快，但代价是 O(n log n) 的排序开销。能不能一遍扫描就搞定， 且不破坏原数组？

⚡ 第三步：用哈希表「边走边记」

换个思路：扫到 nums[i] 时，我们只关心一件事—— 这个数之前出现过吗？

这里要请出一个很好用的工具——哈希表（hash table）。 你可以把它想象成一本「能一眼查到」的号码簿：普通查字典要一页页翻， 但哈希表能几乎一步到位地告诉你「这个东西在不在里面」， 这个查找速度就是 O(1)——不管里面存了多少东西，查一次都差不多快。 因为这道题我们不需要记「这个数对应什么」，只需要记「这个数有没有出现过」， 所以用的是哈希表的简化版——哈希集合（HashSet）， 它就像一个只装「有还是没有」的小本子，不允许记重复的东西。

于是边走边把见过的数记进哈希集合。到一个新数就先查一下： 它在不在集合里？在 → 说明这个数之前已经出现过，也就是撞上了重复， 立刻返回 true；不在 → 说明是第一次见到它，把它记进去，继续往后走。 因为查表几乎不花时间（O(1)），整个数组只需要扫一遍（O(n)）就够了。

当前 nums[i] 已记入哈希表 撞上的「第一次出现」

试试把数组改成 1,2,3,4（无重复）或 1,1,1,3,3,4,3,2,4,2， 看动画分别怎么演示 false 和 true。

☕ 第四步：写成 Java 代码

class Solution {
    public boolean containsDuplicate(int[] nums) {
        // seen：记录「见过的数」
        Set<Integer> seen = new HashSet<>();
        for (int x : nums) {
            // add 返回 false 说明 x 已经在集合里了 —— 撞上了
            if (!seen.add(x)) {
                return true;
            }
        }
        return false;
    }
}
时间复杂度 O(n)

数组只扫一遍，哈希集合的查询 / 插入都是均摊 O(1)。

空间复杂度 O(n)

最坏情况（没有重复）要把所有元素都存进哈希集合。

🚀 小结

套路：当你发现「需要知道某个值之前出没出现过」时， 先想想能不能把见过的东西记进哈希表，把「查找」从 O(n) 变成 O(1)。 这是和两数之和一样的「空间换时间」套路，也是后面去重、计数类题目的通用起点。
