链表只能从头往后走，没法像数组那样直接「首尾互换」。 看完你会明白：用 prev、cur 两个指针一趟扫过去，把每个结点的 next 箭头逐个掉头，链表就反过来了。

🧩 第一步：先搞懂「链表」是什么，题目要什么

你大概见过数组：一排数字挨个排在一起，还编了号（下标），想找第 5 个， 直接跳到下标 5 就行。链表（linked list）不是这样—— 它更像一串手拉手站队的珠子：每颗珠子我们叫它一个结点（node）， 每个结点只装两样东西：自己的值，和一个指向下一颗珠子在哪的 箭头（这个箭头在代码里就叫 next，专业说法是「指针」，你可以就理解成 「手里攥着下一个人的手」）。链表里的结点在内存里可能东一个西一个，完全靠这根 next 箭头才能一个接一个串成一条线——想找第 5 个，只能从第一个开始顺着箭头数过去， 没法像数组那样直接「跳」过去。

题目给你链表的头结点 head（也就是这串珠子的第一颗）， 要你把整条链表的方向反过来，并返回反转后链表的新头结点 （也就是原来的最后一颗珠子）。

head[1, 2, 3, 4, 5]

原来是 1 → 2 → 3 → 4 → 5 → null，反转后变成 5 → 4 → 3 → 2 → 1 → null。新的头结点是原来的最后一个结点。

如果是数组，反转很简单：开两个下标从两头往中间交换值就行，因为数组能直接用下标 「跳」到任意位置。但链表做不到这一点——它没有下标，每个结点只攥着 自己的下一个是谁，完全不知道自己的上一个是谁。 要反转，本质上就是要把每个结点的 next 箭头，从「指向下一个」 改成「指向上一个」。

🐢 第二步：直接改指针会遇到的坑

想法很直接：拿两个变量占位——cur（current，当前正处理的结点）和 prev（previous，它前面那个结点，一开始还没有，就是空的）。 把 cur.next 改成指向 prev 不就行了？但试着写一下就会发现一个坑——

坑在哪：一旦执行 cur.next = prev， cur 原来牵着的下一颗珠子是谁，这条线索就找不回来了—— 因为整条链表全靠 next 这根箭头才能往后走，箭头一改方向， 后面那一大串结点就跟当前处理到的这一段「断联」了，没法继续往下走。

关键洞察：在改指针之前，先用一个临时变量 next 把 cur.next（也就是「下一颗珠子在哪」这条线索）偷偷存一份， 改完指针之后再用这个存好的 next 继续往后走，就不会丢链了。

⚡ 第三步：prev、cur 双指针一趟扫描

用两个指针 prev（一开始是 null）和 cur （一开始是 head）从头到尾扫一遍，每到一个结点重复同样的三个动作：

  • ① 暂存：next = cur.next，先记下原来的下一个结点。
  • ② 掉头：cur.next = prev，让当前结点指向它原来的前一个。
  • ③ 前移：prev = cur，cur = next，两个指针一起往后挪一位。

重复这三步，直到 cur 变成 null（说明已经走完整条链表）。 这时 prev 正好停在原来的最后一个结点上 —— 它就是反转后的新头结点，直接返回它。

prev 指针 / 已反转的箭头 cur 指针（当前处理的结点） 已经掉头完成的结点

试试把链表改成只有 1 一个结点，或者留空表示空链表 []， 看看边界情况下动画怎么处理。

☕ 第四步：写成 Java 代码

下面先给出最常用的迭代写法——就是上面 prev/cur 那一套循环，直接照抄三步口诀。

另外再给一种递归写法：递归说的是一个函数在解决问题时， 先把问题变得更小一点，转手「调用自己」去解决这个更小的问题，等这个更小问题的答案回来了， 再拿这个答案拼出当前问题的答案——就像剥洋葱一层层剥到最里面，再一层层把结果套回去， 也像俄罗斯套娃，一个套一个。对反转链表来说：递归写法假设「从 head.next 往后的那一段已经被反转好了」（这一步交给递归自己去处理），我们只需要操心 自己这一个结点要怎么接到反转好的那一段后面。 

// Definition for singly-linked list.
class ListNode {
    int val;
    ListNode next;
    ListNode() {}
    ListNode(int val) { this.val = val; }
    ListNode(int val, ListNode next) { this.val = val; this.next = next; }
}

class Solution {
    // 写法一：迭代，O(n) 时间、O(1) 额外空间
    public ListNode reverseList(ListNode head) {
        ListNode prev = null;
        ListNode cur = head;

        while (cur != null) {
            ListNode next = cur.next; // ① 先暂存下一个结点
            cur.next = prev;             // ② 把当前结点的指针掉头
            prev = cur;                  // ③ prev、cur 一起前移
            cur = next;
        }
        return prev; // cur 为 null 时，prev 正好停在新头结点上
    }

    // 写法二：递归，O(n) 时间、O(n) 递归栈空间
    public ListNode reverseListRecursive(ListNode head) {
        if (head == null || head.next == null) {
            return head; // 空链表或只剩一个结点，它自己就是新头结点
        }
        ListNode newHead = reverseListRecursive(head.next); // 先反转后面的部分
        head.next.next = head; // 让后面那个结点反过来指向自己
        head.next = null;     // 自己变成新的尾部，断开原来的 next
        return newHead;
    }
}
时间复杂度 O(n)

不管迭代还是递归，链表都只需要遍历一遍。

空间复杂度 迭代 O(1) / 递归 O(n)

迭代只用了 prev、cur、next 三个指针，空间几乎不涨；递归每往下调用自己一层， 就要在调用栈里多摞一层——调用栈你可以想象成一摞盘子， 函数每调用自己一次就往上摞一个新盘子，这次调用结束了才能从最上面拿走（后放的先拿走）。 链表多长，就要摞多高，所以是 O(n)。

🚀 小结

套路：链表上要「反转方向」，记住三步口诀—— 先存下一个、再掉头当前、最后一起前移。 用 prev 记住走过的路，用临时变量 next 保住没走的路， 就不会出现「指针一改、后面全丢」的坑。这套 prev/cur 双指针思路， 后面反转链表的一部分、按组反转、回文链表等题都会反复用到。
