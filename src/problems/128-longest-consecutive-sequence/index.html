数组本身是乱序的，但如果先把它「装进一个能瞬间查找的口袋」， 找连续段就不必排序，一遍扫描就能找到最长的那一串。

🧩 第一步：题目到底要什么

给你一堆乱序的整数，让你找出「数值上连续」的最长一串有多长——比如 1,2,3,4 就是连续的 4 个数。注意：这些数在原数组里的位置可以随便乱放， 只看它们的大小是不是挨着（差 1）。

nums[100, 4, 200, 1, 3, 2]
输出4

把这些数从小到大排一下：1, 2, 3, 4, 100, 200。其中 1、2、3、4 正好连续，长度为 4， 比单独的 100、单独的 200 都长，所以答案是 4。

题目还专门加了一个要求：时间复杂度必须是 O(n)。这是个重要信号—— 它在提醒我们「先排序」这条路其实走不通。

🐢 第二步：直觉的办法为什么不够快

最容易想到的办法：把数组排序，排好序之后连续的数就会挨在一起， 从头扫一遍，数出最长的连续段就行了。这个思路是对的，但排序本身就要 O(n log n)，没法满足题目要求的 O(n)。

瓶颈在哪：我们真正需要反复做的操作，其实只有一件事—— 「判断某个数在不在这堆数里」。排序是为了方便做这件事，但代价太大。

其实有个更直接的工具，叫哈希集合（HashSet）—— 你可以把它想成一本「神奇的查号簿」：往里面「记一个数」几乎不花时间， 之后再问它「某个数在不在里面」，它也能立刻告诉你答案， 不用像翻书那样一页一页找、也不用先排好序再找。这就是文章开头说的 「装进一个能瞬间查找的口袋」——查一次的时间几乎跟口袋里有多少个数无关， 这在算法里记作 O(1)，比排序需要的 O(n log n) 划算多了。

换个思路：先把所有数丢进这个哈希集合（口袋）里。对集合里的每个数 n，往后想 n+1、n+2…… 在不在集合里，能数多远就数多远，记录最长的一段。

但这样还有个隐患：如果对每一个数都往后数一遍，就会有大量重复劳动。 比如 1,2,3,...,10000 这种全连续的数组：从 1 出发要一路数到 10000， 数了将近一万步；可如果紧接着又从 2 出发，再数一遍 2 到 10000，几乎是同样的路白走一趟； 从 3、从 4……每个数都再走一次几乎一样的路，加起来操作次数就变成了 O(n²)—— 又退回到跟暴力排序差不多慢的处境。

关键的优化只有一句话：只从「真正的起点」开始数，其他数一律跳过不数。 怎么判断一个数 n 是不是某段连续序列真正的起点呢？ 只需要看比它小 1 的那个数——也就是 n-1——在不在集合里： 如果 n-1 在集合里，说明 n 前面还接得上别的数， n 肯定不是这一段的开头，直接跳过它，把「往后数」这件事留给 真正的起点去做就行，不用重复劳动。这样一来，每个数一辈子最多只会在 「往后数」的循环里被访问一次，总操作次数还是 O(n)，不会退化。

⚡ 第三步：哈希集合 + 只从起点数

下面这个动画把数组先去重（去掉重复的数）放进哈希集合，然后按从小到大的顺序检查每个数： 有「前驱」（就是比它小 1 的那个数 n-1，在集合里能查到）的， 说明它前面还接得上，不是起点，直接跳过；没有前驱的就是真正的起点， 从它开始一路 +1 往后数，数到数不动（也就是 +1 之后的数不在集合里）为止， 顺便刷新目前为止最长的记录。

☕ 第四步：写成 Java 代码

class Solution {
    public int longestConsecutive(int[] nums) {
        // 先去重放进哈希集合，O(1) 查找某个数在不在
        Set<Integer> set = new HashSet<>();
        for (int n : nums) {
            set.add(n);
        }

        int best = 0;
        for (int n : set) {
            // 只有 n-1 不在集合里，n 才是某段连续序列的起点
            if (!set.contains(n - 1)) {
                int cur = n;
                int len = 1;
                while (set.contains(cur + 1)) {
                    cur++;
                    len++;
                }
                best = Math.max(best, len);
            }
        }
        return best;
    }
}
时间复杂度 O(n)

建集合 O(n)；外层 for 遍历集合每个数一次；内层 while 只有「真正的起点」才会触发， 每个数总共只会被「往后数」访问一次，整体仍是 O(n)。

空间复杂度 O(n)

哈希集合最坏情况要存下数组里所有不同的数。

🚀 小结

套路：当题目要求的复杂度低于「排序」能给到的 O(n log n) 时， 先想想真正需要的操作是不是只是「查找某个值在不在」——哈希集合能把这件事压到 O(1)。 再叠加一个「只从起点开始展开」的小技巧，避免对同一段数据重复劳动， 就能把总操作量真正控制在 O(n)。这种「哈希加速查找 + 只算一次」的组合， 在很多需要判断元素关系的题里都很常见。
