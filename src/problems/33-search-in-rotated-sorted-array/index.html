数组被「切了一刀」整体旋转，看起来乱了，但只要在心里多问一句 ——「现在这一半到底有没有序」，二分查找照样能用，O(log n) 照样拿下。

🧩 第一步：题目到底在说什么

先说清楚两个小词。下标指的是数字在数组里排第几位，从 0 开始数—— 比如 [10,20,30] 里，20 的下标是 1。升序排列就是从左到右 一个比一个大，像 [0,1,2,4,5,6,7] 这样规规矩矩。

现在把一个本来升序排列的数组，从中间「掰断」，前后两段调了个个儿—— 这就叫「旋转」。比如上面那个 [0,1,2,4,5,6,7] 从下标 4 处旋转 （也就是把从下标 4 开始的后半段整体挪到最前面），就变成了 [4,5,6,7,0,1,2]。现在给你这个旋转后的数组和一个 target， 要你找到它的下标；找不到就返回 -1。

nums[4,5,6,7,0,1,2]
target0

target = 0 在下标 4 上 → 返回 4。题目还要求时间复杂度必须是 O(log n)，这就把「从头扫到尾」的办法直接排除了。

🐢 第二步：为什么不能直接二分

最笨的办法是把数组从头到尾挨个看一遍，看有没有哪个正好等于 target， 最坏情况要看完全部 n 个数，这就是 O(n)——能做出来， 但题目明确要求 O(log n)，这条路走不通。

O(log n) 是什么概念？如果数组本来就整体有序 （从左到右一路升序），有一种经典办法叫「二分查找」： 先看数组正中间那个数 nums[mid]——如果它比 target 大， target 就只可能藏在左边那一半；如果它比 target 小，target 就只可能藏在右边那一半。 每比一次，能找的范围就直接砍掉一半。这就像猜数字游戏：主持人心里想好 1～100 里的一个数，你每猜一次，他就告诉你「猜大了」还是「猜小了」，你按提示不断把范围对半砍， 最多猜 7 次左右就能锁定答案——这比从 1 数到 100 快多了，这也是「log」的含义： 砍一次范围缩一半，要砍 log₂n 次左右才能缩到只剩一个数。 这里的 lo、hi 就是当前还没被排除掉的搜索范围的 左边界和右边界（low / high 的缩写）， mid 是这段范围正中间的下标。

但普通二分查找有个前提：数组必须整体有序。而旋转数组整体看是「乱」的—— 比如 [4,5,6,7,0,1,2]，直接拿中间的 nums[mid] 和 target 比大小，完全判断不出该往左走还是往右走：target 可能藏在左边那截「高位数」里， 也可能藏在右边那截「低位数」里。

关键洞察：虽然整个数组不是有序的，但用 mid 把它切成两半后， 至少有一半仍然是严格升序的。只要先看出哪一半有序， 再判断 target 是否落在这段有序区间里，就总能把搜索范围砍掉一半—— 这正是二分查找「每次砍一半」的精神，只是多了一步「先认出哪边规矩」。

⚡ 第三步：带「先判断哪半有序」的二分

每一步先取 mid = (lo + hi) / 2，分三种情况：

  • nums[mid] == target：直接命中，返回 mid。
  • 左半 [lo, mid] 有序（即 nums[lo] <= nums[mid]）： 看 target 是否落在 [nums[lo], nums[mid]) 里—— 是就去左半找（hi = mid - 1），不是就去右半找（lo = mid + 1）。
  • 否则右半 [mid, hi] 有序： 看 target 是否落在 (nums[mid], nums[hi]] 里—— 是就去右半找（lo = mid + 1），不是就去左半找（hi = mid - 1）。
判断「左半是不是有序」时用的是 nums[lo] <= nums[mid] （带等号），这是特意留的：当区间缩到只剩一个数时 lo 会等于 mid，这时候左边等于右边，也该算作「有序」，不然会漏判这种情况。

拿 [4,5,6,7,0,1,2]、target = 0 走一遍，感受一下每一步是怎么砍半的：

  • lo=0, hi=6，mid=3，nums[mid]=7。 nums[lo]=4 <= nums[mid]=7，左半 [4,5,6,7] 有序。 target=0 不在 [4,7) 里 → 去右半，lo=mid+1=4。
  • lo=4, hi=6，mid=5，nums[mid]=1。 nums[lo]=0 <= nums[mid]=1，左半 [0,1] 有序。 target=0 落在 [0,1) 里 → 去左半，hi=mid-1=4。
  • lo=4, hi=4，mid=4，nums[mid]=0 正好等于 target → 命中，返回下标 4。

不管哪种情况，每一步都能确定地排除一半，区间指数级收缩，整体 O(log n)。

区间边界 lo / hi 当前检查 mid 本步判定为「有序」的那一半 命中 target

☕ 第四步：写成 Java 代码

class Solution {
    public int search(int[] nums, int target) {
        int lo = 0, hi = nums.length - 1;
        while (lo <= hi) {
            int mid = lo + (hi - lo) / 2;
            if (nums[mid] == target) {
                return mid;
            }
            if (nums[lo] <= nums[mid]) {
                // 左半 [lo, mid] 有序
                if (nums[lo] <= target && target < nums[mid]) {
                    hi = mid - 1; // target 在左半里
                } else {
                    lo = mid + 1; // target 在右半里
                }
            } else {
                // 右半 [mid, hi] 有序
                if (nums[mid] < target && target <= nums[hi]) {
                    lo = mid + 1; // target 在右半里
                } else {
                    hi = mid - 1; // target 在左半里
                }
            }
        }
        return -1;
    }
}
时间复杂度 O(log n)

每一步都能排除一半区间，和普通二分查找一样的收敛速度—— 比如 nums 长度有一百万，最多也就二十步左右能锁定 target，比逐个扫描快得多。

空间复杂度 O(1)

只用了 lo / mid / hi 几个变量。

🚀 小结

套路：看到「局部有序」或者「被切了一刀的有序数组」，先别急着否定二分查找—— 每次用 mid 把数组切开，总有一半是规规矩矩升序的， 先认出这一半，再用普通的「target 是否在区间内」去判断走哪边， 二分查找的框架完全不用变，只是多了一步「先看哪边有序」的判断。
