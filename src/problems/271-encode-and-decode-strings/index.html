这是一道经典的「设计」题：把一组字符串打包成一条字符串发出去，再原样拆回来。 难点不在打包，而在拆包时怎么知道每一段在哪结束—— 哪怕字符串里混着逗号、井号，甚至本身就是空串。

🧩 第一步：题目到底要什么

你要实现两个函数：encode(strs) 把一个字符串数组打包成 一整条字符串；decode(s) 再把这条字符串原样拆回 和原数组一模一样 的字符串数组（包括顺序、包括空串）。

strs["am#1", "co,de", "", "ab"]

注意这几个字符串本身就带着「逗号」「井号」和「空串」——这些恰好是我们最容易拿来当分隔符的字符， 题目就是故意来挑战你的编码方案够不够稳。

🐢 第二步：直觉做法为什么会翻车

最容易想到的：用逗号把字符串拼起来，比如 "am#1,co,de,,ab"，解码时按逗号 split 回去。看起来简单，但马上就会出事。

分隔符会被「内容」冒充： 原数组里第二个字符串是 "co,de"，它自己就含逗号。拼接后变成 ...,co,de,...，解码时一 split，"co,de" 被拦腰切成了 "co" 和 "de" 两段——结果就错了。换成 # 当分隔符也一样会被 "am#1" 坑。只要分隔符可能出现在字符串内容里，纯「找分隔符」的思路就靠不住。

⚡ 第三步：给每段「标好长度」，不用再找分隔符

换个思路：与其让解码方去找每段在哪结束，不如编码时就直接告诉它 每段有多长。约定格式：长度 + # + 内容，例如长度为 2 的 "ab" 编码成 "2#ab"，把所有片段首尾拼起来就是整条编码串。

解码时从左往右扫：先一直读数字，直到遇到第一个 #，这串数字就是「接下来要读几个字符」； 然后不多不少地往后取那么多个字符，作为一个还原出的字符串，再接着处理下一段。 因为长度是提前写死的，所以不管这些字符里藏着逗号、#，甚至长度是 0（空串）， 都不会读错一个字符——我们是「按长度跳」，而不是「找分隔符」。

当前正在读写的片段 已确定/已写入 长度前缀里的 #

试试把输入改成含逗号、# 或空串的字符串（用 | 分隔）， 看长度前缀编码怎么把它们都安全地还原回来。

☕ 第四步：写成 Java 代码

public class Codec {

    // 编码：每个字符串前面写「长度 + #」，再拼上字符串本身
    public String encode(List<String> strs) {
        StringBuilder sb = new StringBuilder();
        for (String s : strs) {
            sb.append(s.length()).append('#').append(s);
        }
        return sb.toString();
    }

    // 解码：先读数字直到遇到 #，拿到长度 L，再不多不少地往后取 L 个字符
    public List<String> decode(String s) {
        List<String> res = new ArrayList<>();
        int i = 0;
        while (i < s.length()) {
            int j = i;
            while (s.charAt(j) != '#') {
                j++; // 扫到 # 为止，[i, j) 就是长度数字
            }
            int len = Integer.parseInt(s.substring(i, j));
            String str = s.substring(j + 1, j + 1 + len); // 按长度精确截取，不怕里面有逗号/#
            res.add(str);
            i = j + 1 + len; // 指针跳到下一段开头
        }
        return res;
    }
}
时间复杂度 O(n)

编码、解码都只各自把所有字符扫一遍，n 是所有字符串长度之和。

空间复杂度 O(n)

编码结果和解码出的字符串总长度都是 O(n)。

🚀 小结

套路：但凡要把「不定长、可能含任意字符」的内容拼接后再无歧义地拆开， 别指望靠挑一个「看起来不常用」的分隔符——内容里永远可能藏着它。 稳妥的办法是自描述长度：先告诉对方接下来这一段有多长， 对方按长度精确截取，根本不需要「找」分隔符。这个套路在序列化、网络协议（如 HTTP 的 Content-Length）里随处可见。
