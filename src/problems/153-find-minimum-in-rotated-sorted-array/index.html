数组被「掰断」旋转之后，最小值其实就藏在那个「断点」上。 不用挨个扫，二分法照样能把它在 O(log n) 内揪出来。

🧩 第一步：题目到底在说什么

想象你手里有一副已经从小到大排好的牌，比如 [0,1,2,4,5,6,7]。 现在有人从中间「掰断」，把后面一截整个搬到最前面——比如搬 4 张过去， 就变成了 [4,5,6,7,0,1,2]。这就是题目说的「旋转」。仔细看会发现： 前半段 4,5,6,7 自己是升序的，后半段 0,1,2 自己也是升序的， 只是整体看起来像「一路变大、突然掉下来一大截、再接着变大」。

现在只给你旋转后的数组（保证里面的数字互不相同）， 要你找出其中的最小值是多少。

nums[4,5,6,7,0,1,2]

最小值 0 就藏在「从大突然掉到小」的那个断口位置，也就是下标 4 （下标就是数组里每个位置从 0 开始编的号，0 号、1 号……一直往后数）。 题目要求做到 O(log n)，也就是要比「从头到尾挨个扫一遍」的 O(n) 快得多——数组越长，这个差距越明显。

🐢 第二步：先搞懂「二分查找」是什么，再看它为什么不能直接照搬

如果你没接触过算法，先花十秒钟理解一个概念——二分查找， 说白了就是玩「猜数字」游戏：你心里想一个 1~100 的数，我每次只猜一个数， 你告诉我「大了」还是「小了」，我就能把还可能的范围砍掉一半，很快就能猜中， 比从 1 挨个往上猜快得多。放到一个排好序的数组里也是同样的道理： 每次看一眼区间正中间那个数，跟目标比一比大小，就知道该往左边找还是往右边找， 同时把候选范围砍掉一半——这就是「二分」两个字的来历。

但这道题没法直接照搬「猜数字」的模板，原因有两点：第一，我们不是在找某个「给定的目标值」， 而是要找数组里的最小值本身，压根没有东西可比大小；第二，也是更麻烦的一点—— 旋转之后的数组整体已经不是升序了（就像上面例子里那样，先变大再突然掉下来）， 普通二分查找要求数组整体有序，没法简单地拿 nums[mid] 去和某个固定值比大小来判断该往哪边走。

当然，最笨的办法依然是从头到尾扫一遍找最小值，O(n)，能算出正确答案， 但完全没用上「数组其实是两段分别有序拼起来的」这个特点，也不满足题目 O(log n) 的要求。我们需要换一种「每次砍掉一半」的判断方法。

关键洞察：旋转数组其实是两段「分别升序」的子数组拼起来的： 前一段（旋转点之前）的所有值，都大于等于后一段（旋转点之后）的所有值。 那个「旋转点」——也就是从大突然掉到小的那个位置——正好就是最小值所在的下标。 只要每次拿 nums[mid] 和 nums[hi]（当前区间最右边那个数） 比一比，就能判断 mid 落在「较大」的前段还是「较小」的后段，从而知道最小值在 mid 的哪一侧—— 这依然是「每次砍掉一半」的二分思路，只是判断方向的方法从「跟目标比」换成了「跟右边界比」。

⚡ 第三步：用「nums[mid] vs nums[hi]」收缩区间

用两个指针 lo 和 hi（low / high 的缩写，就是「当前还没被排除掉、 可能藏着最小值」的这段区间的左右边界）圈出一段范围：一开始 lo 指向数组开头， hi 指向数组结尾。每一步取区间正中间的位置 mid = lo + (hi - lo) / 2（这样写和更直觉的 (lo + hi) / 2 算出来的结果一样，只是很多程序员习惯这么写，可以避免某些语言里 lo + hi 数值过大时相加溢出——这题数据量不大，不用担心，看到这个写法认识它是干嘛的就好）， 只比较 nums[mid] 和 nums[hi]：

  • nums[mid] > nums[hi]：说明 mid 还在「靠前、较大」的 那一段里，断点（最小值）一定在 mid 右边（mid 本身不可能是答案） → lo = mid + 1。
  • nums[mid] <= nums[hi]：说明 mid 已经进入「靠后、较小」 的那一段（或者整个数组根本没乱序），最小值在 mid 或 mid 左边 （mid 自己仍然是候选）→ hi = mid（注意不是 mid - 1， 因为 mid 还可能就是答案）。

这样 [lo, hi] 每一步至少缩小一半，直到 lo == hi—— 也就是区间里只剩一个下标，没得再筛了——这个重合的下标就是最小值所在的位置。 因为每一步都能把区间砍掉一半，所以只需要大约 log n 步就能收敛， 整体时间复杂度是 O(log n)。

区间边界 lo / hi mid 仍是候选（保留） mid 被排除（不可能是最小值） 最终确定的最小值

试试把数组改成 11,13,15,17（其实没旋转，n=4，rotated n 次）， 看二分依旧能正确收敛到下标 0。

☕ 第四步：写成 Java 代码

class Solution {
    public int findMin(int[] nums) {
        int lo = 0, hi = nums.length - 1;
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (nums[mid] > nums[hi]) {
                // mid 在「较大」那一段，最小值在 mid 右边
                lo = mid + 1;
            } else {
                // mid 已在「较小」那一段，最小值是 mid 或更靠左，保留 mid
                hi = mid;
            }
        }
        return nums[lo]; // lo == hi，就是最小值所在下标
    }
}
时间复杂度 O(log n)

区间每一步至少减半，和普通二分查找一样的收敛速度。

空间复杂度 O(1)

只用了 lo / mid / hi 几个变量。

🚀 小结

套路：旋转有序数组的最小值，本质上是在找「从大突然掉到小」的那个断点。 二分时不需要判断「哪一半有序」这种复杂逻辑，只需要把 nums[mid] 和 右边界 nums[hi] 比一比，就能确定断点在 mid 的左边还是右边—— 这个「跟右边界比」的小技巧，在很多旋转数组类问题里都能直接套用。
