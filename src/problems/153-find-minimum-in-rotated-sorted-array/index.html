数组被「掰断」旋转之后，最小值其实就藏在那个「断点」上。 不用挨个扫，二分法照样能把它在 O(log n) 内揪出来。

🧩 第一步：题目到底在说什么

一个本来升序排列的数组，被整体旋转了若干次：把末尾的几个数搬到了最前面。 比如 [0,1,2,4,5,6,7] 旋转 4 次，就变成了 [4,5,6,7,0,1,2]。 现在只给你旋转后的数组（保证所有数互不相同），要你找出其中的最小值。

nums[4,5,6,7,0,1,2]

最小值 0 就藏在「从大到小那个断口」的位置（下标 4）。题目要求 O(log n)， 从头扫到尾的 O(n) 办法不满足要求。

🐢 第二步：为什么不能直接二分

最笨的办法是从头到尾扫一遍找最小值，O(n)，能过但不满足题目要求。 普通二分查找要求数组整体有序，而旋转数组整体看是「先高后低」的—— 不能直接拿 nums[mid] 去和某个固定值比大小来判断该往哪边走。

关键洞察：旋转数组其实是两段「分别升序」的子数组拼起来的： 前一段（旋转点之前）的所有值，都大于等于后一段（旋转点之后）的所有值。 那个「旋转点」——也就是从大突然掉到小的那个位置——正好就是最小值所在的下标。 只要每次拿 nums[mid] 和 nums[hi] 比一比，就能判断 mid 落在哪一段，从而知道最小值在 mid 的哪一侧。

⚡ 第三步：用「nums[mid] vs nums[hi]」收缩区间

维护区间 [lo, hi]，每一步取 mid = lo + (hi - lo) / 2， 只比较 nums[mid] 和 nums[hi]：

  • nums[mid] > nums[hi]：说明 mid 还在「靠前、较大」的 那一段里，断点（最小值）一定在 mid 右边（mid 本身不可能是答案） → lo = mid + 1。
  • nums[mid] <= nums[hi]：说明 mid 已经进入「靠后、较小」 的那一段（或者整个数组根本没乱序），最小值在 mid 或 mid 左边 （mid 自己仍然是候选）→ hi = mid（注意不是 mid - 1， 因为 mid 还可能就是答案）。

这样 [lo, hi] 每一步至少缩小一半，直到 lo == hi， 这个重合的下标就是最小值的位置，整体 O(log n)。

区间边界 lo / hi mid 仍是候选（保留） mid 被排除（不可能是最小值） 最终确定的最小值

试试把数组改成 11,13,15,17（其实没旋转，n=4，rotated n 次）， 看二分依旧能正确收敛到下标 0。

☕ 第四步：写成 Java 代码

class Solution {
    public int findMin(int[] nums) {
        int lo = 0, hi = nums.length - 1;
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (nums[mid] > nums[hi]) {
                // mid 在「较大」那一段，最小值在 mid 右边
                lo = mid + 1;
            } else {
                // mid 已在「较小」那一段，最小值是 mid 或更靠左，保留 mid
                hi = mid;
            }
        }
        return nums[lo]; // lo == hi，就是最小值所在下标
    }
}
时间复杂度 O(log n)

区间每一步至少减半，和普通二分查找一样的收敛速度。

空间复杂度 O(1)

只用了 lo / mid / hi 几个变量。

🚀 小结

套路：旋转有序数组的最小值，本质上是在找「从大突然掉到小」的那个断点。 二分时不需要判断「哪一半有序」这种复杂逻辑，只需要把 nums[mid] 和 右边界 nums[hi] 比一比，就能确定断点在 mid 的左边还是右边—— 这个「跟右边界比」的小技巧，在很多旋转数组类问题里都能直接套用。
