不让用 + 和 -，加法还能怎么做？ 这道题逼你把「加法」拆开来看：原来它从来就是「异或算和、与并左移算进位」反复来回， 这正是计算机里加法器真正的工作原理。

🧩 第一步：题目到底要什么

给你两个整数 a 和 b，要你算出它们的和——但不准用 +，也不准用 -。也就是说，平时写 a + b 这一个字符就能搞定的事，现在只能靠 按位运算（&、|、^、 <<、>>）自己「拼」出一套加法逻辑。

先补一点背景知识，不然后面全是天书：我们平时写的数字（比如 5、12）是 十进制——逢十进一，用 0~9 这十个符号。但计算机里存的整数， 骨子里全是二进制——逢二进一，只有 0 和 1 两个符号。比如十进制的 5，在计算机里其实存的是 101； 十进制的 3，存的是 011。「按位运算」 就是把两个数的二进制一位一位对齐，然后每一位单独做某种运算—— 先不管进位的事，进位是另外单独算出来再加上去的。下面几步会带你一步步看清楚， 这些陌生符号到底在算什么。

a1
b2

1 和 2 的二进制分别是 01 和 10，正好每一位都不同、 没有任何一位「撞 1」，加起来不会产生进位，直接拼成 11 = 3。

💡 第二步：拆开「手算加法」到底在干什么

回想小学竖式加法，比如 5 + 7：你会先把每一位本身加一加， 算不动的地方（比如个位 5+7=12，写 2 进 1）就单独「进位」到高一位， 然后再把进位加上去——如果加完又产生新的进位，就继续往上进，直到不再有进位为止。 换句话说，「加法」其实一直都能拆成两件独立的事：① 每一位自己先加、 ② 把进位单独搬到高一位再加一次。二进制下，这两件事都能直接用 位运算精确表达出来，不需要 + 也不需要 -：

① 无进位和 = a ^ b： ^ 叫「按位异或」（读作 XOR），意思是把 a、b 的二进制一位一位对齐， 同一位上两个数不同就得 1，相同就得 0。 这恰好就是「完全不考虑进位」时，这一位单独相加该有的结果 （0+1=1、0+0=0、1+1 本来该是 2，但先不进位、只留本位的话就是 0）。
② 进位 = (a & b) << 1： & 叫「按位与」，只有两个数同一位都是 1，这一位算出来才是 1，也就是「才会产生进位」的那些位； << 1 是「左移一位」——把这一串二进制数集体往左挪一格、 右边补个 0（相当于整体乘以 2）。之所以要左移，是因为进位不能留在原地， 它必须加到高一位上去，跟竖式加法里「进 1」要写到左边那一位是同一个道理。

于是套路就是：先算一次「无进位和」和「进位」，让 a 变成无进位和、 b 变成进位，再重复同样的两步——直到某一轮算出的进位 b 变成 0（没有新的进位可加了），这时候的 a 就是最终答案。Java 的 int 是用固定 32 个 0/1 表示的， 所以这个过程最多循环 32 轮就一定会结束。

⚡ 第三步：逐轮「异或 + 进位」动画

下面把 a、b 都按 32 位二进制展开——也就是说， 每个数都画成一排 32 个格子，每格填 0 或 1。每一轮拆成两小步： 先异或出「无进位和」，再与并左移算出「进位」，算完就把 a、b 更新成这两个新值，继续判断 b 是否归零。

负数怎么办：如果试着把 a 改成负数，你会看到它的 32 格里最左边一大片 都是 1。这是因为计算机表示负数用的是一种叫补码的规则： 正数就是它本来的二进制；负数则是先把对应正数的每一位「取反」（0 变 1、1 变 0）， 再在最后加 1。你不需要背这个规则，只要记住结论就够了：负数在计算机里也是实实在在的 一串 0/1，最高位（最左边）通常是 1，本题的按位运算对正数、负数是同一套逻辑， 完全不用特殊处理。
a/b 中「不同」的位（异或阶段） a/b 中「都是 1」的位（进位阶段） 本步算出的结果中为 1 的位

试试把 a 改成 -1、b 改成 1，看看负数的补码在最高位「全是 1」 时，要绕多少轮进位才能把它们一点点抵消干净。

☕ 第四步：写成 Java 代码

class Solution {
    public int getSum(int a, int b) {
        // b 表示「还没加进 a 里的进位」，b 变成 0 时 a 就是最终结果
        while (b != 0) {
            int carry = (a & b) << 1; // 进位：同一位都是 1 才会进位，左移到高一位
            a = a ^ b;                  // 无进位和：按位异或
            b = carry;                  // 把进位留到下一轮继续加
        }
        return a;
    }
}
时间复杂度 O(1)

固定在 32 位整数范围内运算，循环次数被位宽（32）卡死上限，与数值大小无关。

空间复杂度 O(1)

只用了几个整型变量，不需要额外数组或哈希表。

🚀 小结

套路：但凡看到「不准用 +/-」这类限制， 基本都是在考「加法的位运算实现」——记住这对黄金搭档： 异或 算「这一位单独相加的结果」，与 + 左移 算「要进到高一位的量」， 反复迭代直到进位为 0。这套「异或/与」的拆解思路不止能做加法， 后面遇到「不用额外变量交换两数」「找数组中只出现一次的数字」之类的位运算题， 本质上都是同一套工具箱在换着花样用。
