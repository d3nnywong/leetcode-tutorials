一个会「举一反三」的单词本：不仅能记住你加进去的单词，还能用 . 当万能字符去模糊查找。 看完你会明白：把单词的公共前缀拼成一棵树（字典树 / Trie），再配上一点点回溯，就是答案。

🧩 第一步：题目到底要什么

要设计一个 WordDictionary：可以不断 addWord(word) 往里面加单词， 也可以 search(word) 查一个单词在不在里面。特别之处是—— search 的查询词里可能带 .，每个 . 可以匹配 任意一个字母。换句话说，search(".ad") 在问： 「有没有一个三个字母的单词，第一个字母随便、后两个是 ad？」

已添加bad, dad, mad
search".ad"

第一个字符是通配符，bad / dad / mad 随便哪个都能对上后两位 ad → 返回 true。

🐢 第二步：最直接的笨办法，以及关键洞察

最简单的做法：准备一个数组，把每次 addWord 加进来的单词原样存起来 （数组你可以理解成一排格子，一个挨一个存东西，按顺序从 0 开始编号）。 每次 search 就把查询词和数组里每一个单词逐字符比对 （遇到 . 就跳过这一位的比较，两边随便对上什么字母都算过）， 只要有一个单词能全部对上就返回 true。

慢在哪：一次 search 要扫一遍全部单词、每个单词还要逐字符比， 是 O(n · L)（n 是单词数量，L 是单词长度）。题目最多 10⁴ 次调用， 单词之间又有大量重复的前几个字母（比如 bad / dad / mad 都以 ad 结尾、 cat / car / cap 都以 ca 开头），可我们每次都要把这些重复的字母 从头再比一遍，完全是在做无用功。

关键洞察：既然很多单词共享前面几个字母，那能不能把它们「拼」在一起， 相同的部分只存一份？这就要用到树这种结构——你可以把它想成一棵倒过来长的家谱： 最顶上有一个出发点，叫根；根往下分叉出几条「树枝」（专业说法叫「边」）， 每条树枝的末端又能继续往下分叉，一路分到最底下不再分叉的地方。树上每一个分叉点、 每一个末端，都叫一个节点。

把这种树专门用来存单词，就是字典树（Trie）：从根节点出发， 每一条树枝代表一个字母，顺着树枝往下走，走过的字母连起来就是一个前缀。 比如 bad、dad、mad 这三个词，首字母各不相同， 但只要各自走到第二个字母 a，剩下的 d 完全可以共用同一条树枝、 同一个节点——公共部分只占一份空间，不用重复存三次。插入一个单词、或者精确查找 一个不带 . 的单词，都只是顺着字母从根往下走到底，是 O(L)， 跟数据库里到底存了多少个单词完全无关。

那通配符 . 该怎么处理？既然它能匹配任意一个字母，走到这一步时， 就把当前节点手头实际拥有的那几条分支都试一遍 （而不是像暴力法那样，把数据库里全部单词都比一遍）——某条分支走到底发现走不通， 就退回来换下一条试试。这种「一条路走到黑，走不通再回头换一条」的搜索方式， 叫深度优先搜索；「走不通就退回上一步、再尝试别的选择」这个动作本身， 就叫回溯。写代码时，我们会让一个函数在自己内部再调用自己一次， 去处理「剩下还没匹配完的那一小段查询词」——这种函数自己调用自己、去解决一个更小的 同类问题的写法，叫递归，有点像俄罗斯套娃，一层套一层， 但每一层都比上一层小一点，总有一天会套到最里面那个不能再拆的娃娃，递归也就停下来了。

⚡ 第三步：动画演示——插入建树 + 通配符回溯查找

先看字典树是怎么一个单词一个单词「长」出来的：遇到新字母就新长出一个节点， 遇到已经存在的字母就直接复用原来的节点。再看 search 时怎么沿着树往下走—— 普通字母只有唯一一条树枝可走，遇到 . 就把这个节点的每一条分支都试一遍， 走不通就退回来换一条（也就是上一步说的「深度优先 + 回溯」）。

☕ 第四步：写成 Java 代码

代码里的 Node 就是字典树上的一个节点：它有 26 个「子节点」的位置 （对应 a~z 26 个字母，没走到的字母就留 null），还有一个 isEnd 标记「从根节点走到这里，正好拼出了一个完整的单词」。 dfs 就是那个「自己调用自己」的递归函数：碰到普通字母，只有唯一一条路， 直接递归到下一个字符；碰到 .，就对 26 个子节点挨个递归试一遍， 只要有一个能走通，整体就算成功——这正是「深度优先 + 回溯」翻译成代码的样子。 

class WordDictionary {
    // Trie 节点：26 个小写字母的子节点 + 是否是某个单词的结尾
    static class Node {
        Node[] children = new Node[26];
        boolean isEnd = false;
    }

    private final Node root;

    public WordDictionary() {
        root = new Node();
    }

    public void addWord(String word) {
        Node cur = root;
        for (char c : word.toCharArray()) {
            int idx = c - 'a';
            if (cur.children[idx] == null) {
                cur.children[idx] = new Node();
            }
            cur = cur.children[idx];
        }
        cur.isEnd = true; // 标记单词结尾
    }

    public boolean search(String word) {
        return dfs(root, word, 0);
    }

    // 在 node 这棵子树里，从 word 的第 idx 个字符开始匹配
    private boolean dfs(Node node, String word, int idx) {
        if (node == null) return false;
        if (idx == word.length()) return node.isEnd;

        char c = word.charAt(idx);
        if (c != '.') {
            return dfs(node.children[c - 'a'], word, idx + 1);
        }
        // '.' 通配符：把 26 个子节点都试一遍，只要有一个能匹配成功就行
        for (Node child : node.children) {
            if (dfs(child, word, idx + 1)) {
                return true;
            }
        }
        return false;
    }
}
时间复杂度 addWord: O(L)　search: O(L) ~ O(26^d · L)

L 是单词长度。addWord 和不带 . 的 search 都只是顺着一条路径走到底，O(L)。带 . 时最坏要在每个通配符位置分叉， d 是 . 的个数（题目保证 d ≤ 2），所以整体仍然可控。

空间复杂度 O(总字符数 × 26)

每个字符对应一个 Trie 节点，节点里固定开 26 个子节点指针；公共前缀只占一份空间。

🚀 小结

套路：当一堆字符串有大量公共前缀、又要支持「精确匹配 + 模糊匹配」时， 字典树（Trie）几乎是第一反应——用树形结构把共享前缀压成一份， 再用「沿着边走」把字符匹配从「整串比较」降到「逐字符判断」。 遇到通配符这种「有多种可能」的场景，就在树上做一次小小的深度优先 + 回溯， 这个组合在自动补全、IP 路由、单词游戏里都会反复出现。
