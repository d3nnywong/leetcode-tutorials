k 条已经各自排好序的链表，怎么把它们拼成一条更长的有序链表？ 看完你会明白：一个小顶堆，怎么把「每次都要从 k 个候选里挑最小」这件事做到飞快。

🧩 第一步：题目到底要什么

先弄清楚什么是「链表」：它不像数组那样把所有数字整整齐齐挨在一起放， 而更像一串珠子——每一颗珠子（我们管它叫「节点」）只攥着「下一颗珠子在哪」这一个信息。 想从头走到尾，只能顺着这根线一颗一颗往下摸，没法像数组那样直接跳到第 5 颗。 每条链表最前面的那颗珠子，就叫它的头节点。

这道题给你 k 条这样的链表，每一条自己内部都已经是升序的 （珠子上的数字从头到尾从小到大排好了）。要求把它们全部合并成一条链表， 合并后的结果也必须是升序的。

lists[[1,4,5],[1,3,4],[2,6]]

三条链表各自有序：1→4→5、1→3→4、2→6。把它们「拧」成一条： 1→1→2→3→4→4→5→6。注意里面可以有重复值（两个 1）。

如果只有两条链表，这就是经典的「合并两个有序链表」：拿两条链表当前最前面还没用过的那颗珠子比一比， 值小的那颗摘下来接到结果链表末尾，它所在的链表就往前挪一格、露出下一颗珠子，继续比下一轮。 现在的难点是 k 可以很多（题目里最多能到 10⁴ 条）——怎么把这个「比一比、摘一个」的思路， 从 2 条链表扩展到 k 条？

🐢 第二步：直接的笨办法，慢在哪

最容易想到的：把 k 条链表两两合并——先用「比一比、摘一个」的办法 合并链表 1 和链表 2，拼成一条新链表；这条新链表再去和链表 3 合并；合并结果再和链表 4 合并…… 像滚雪球一样，一共依次做 k − 1 次「合并两个有序链表」。

慢在哪：设所有节点总数是 N。每一次「合并两个有序链表」最坏要扫完当前已经合并出的那条 （越往后越长），k − 1 次合并下来，总代价是 O(N·k)。 k 一大（题目里能到 10⁴），就会很慢。

换个角度想：其实我们每一步只关心一件事——k 条链表「当前最前面那颗还没用掉的珠子」里， 谁的值最小。每次把这颗最小的摘下来接到结果末尾，它所在的链表就往前挪一格， 换出一颗新的候选珠子。也就是说，我们需要一个工具：手里最多同时攥着 k 个候选值， 还要能随时很快地告诉我们「这堆里谁最小」，拿走最小的之后再塞进一个新值， 下一次依然能很快找到新的最小值——而不是每次都把 k 个候选重新挨个比一遍。 这正是小顶堆（也叫优先队列）最擅长的事：你可以把它想象成一个会自动 把最小的那个顶到最上面的桶——不管往里扔一个新值，还是从最上面取走一个值， 它都能在 O(log k) 的时间内完成（比逐个比较快得多），并且始终保证最上面是当前最小的。

⚡ 第三步：用小顶堆做 k 路归并

先把每条链表的头节点都放进一个小顶堆（按节点值排序，堆顶永远是当前所有候选里最小的那个）。 然后反复做同一件事：从堆顶弹出最小的节点，接到结果链表末尾；如果它在原来那条链表里还有下一颗珠子， 就把那颗新珠子放进堆里，继续当候选。堆空了，就说明所有珠子都已经按从小到大的顺序接到结果链表上了。

为什么这样一定是对的？因为堆顶弹出来的，永远是「当前所有还没被接走的珠子」里最小的那个—— 不可能存在一颗更小、还没被接进结果链表的珠子，所以每次接进结果链表的顺序天然就是升序的， 不需要事后再检查一遍。

每颗珠子入堆、出堆各恰好一次，堆里最多同时存在 k 个节点 （因为任意时刻每条链表最多只有一颗珠子在堆里候选），所以每次堆操作是 O(log k)， 一共对 N 颗珠子都做一遍，整体就是 O(N log k)——比两两合并的 O(N·k) 快得多。

堆中候选节点（各链表当前最前面的节点） 堆顶最小值（即将弹出） 已接入结果链表的节点

试着改成 5,6|1,2,9|3 或者带一条空链表 1,4,5|， 看看堆是怎么始终维护「每条链表当前候选」并挑出最小值的。

☕ 第四步：写成 Java 代码

Java 自带的 PriorityQueue 就是一个现成的小顶堆，我们只要告诉它 「按节点值从小到大比较」，它就会一直帮我们把当前最小的候选顶在最上面， 随时能用 poll() 弹出、用 offer() 塞入新候选。 另外代码里还用了一个小技巧——哨兵节点 dummy： 它本身不算结果链表里的一颗珠子，只是提前占一个「起点」的位置，方便我们从它后面开始往上接珠子， 这样就不用单独判断「结果链表现在还是空的」这种边界情况；最后返回 dummy.next （跳过这颗假的起点），才是真正拼好的结果链表。 

// 单链表节点定义（LeetCode 已给出）
class ListNode {
    int val;
    ListNode next;
    ListNode() {}
    ListNode(int val) { this.val = val; }
    ListNode(int val, ListNode next) { this.val = val; this.next = next; }
}

class Solution {
    public ListNode mergeKLists(ListNode[] lists) {
        if (lists == null || lists.length == 0) return null;

        // 小顶堆：按节点值从小到大排，堆顶永远是当前最小候选
        PriorityQueue<ListNode> heap =
            new PriorityQueue<>((a, b) -> a.val - b.val);

        // 把每条链表的头节点放进堆（跳过空链表）
        for (ListNode head : lists) {
            if (head != null) heap.offer(head);
        }

        ListNode dummy = new ListNode(-1); // 哨兵节点，简化「结果链表为空」的边界
        ListNode tail = dummy;

        while (!heap.isEmpty()) {
            ListNode min = heap.poll();   // 弹出当前全局最小
            tail.next = min;                  // 接到结果链表末尾
            tail = tail.next;
            if (min.next != null) heap.offer(min.next); // 它所在链表还有下一个，补入堆
        }

        return dummy.next;
    }
}
时间复杂度 O(N log k)

N 是所有节点总数，k 是链表条数。每个节点恰好入堆、 出堆各一次，堆大小不超过 k，单次堆操作 O(log k)。

空间复杂度 O(k)

堆里最多同时存放 k 个节点；不算输出结果链表本身占用的空间。

🚀 小结

套路：当你要从「k 个有序来源」里不断挑出当前最小（或最大）的元素时， 不需要每次都把 k 个候选重新比一遍——用小顶堆（优先队列） 这个「自动把最小值顶上来」的桶，把「挑最小」这件事从逐个比较的 O(k) 降到 O(log k)。这个套路不只适用于合并 k 个有序链表， 也是「外部排序」「Top K」「多路归并」这类问题的通用解法。
