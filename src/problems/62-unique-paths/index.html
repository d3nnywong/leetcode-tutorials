一个机器人只能往下或往右走，从网格左上角走到右下角， 走法的数量就藏在一张「逐格相加」的二维 DP 表里。

🧩 第一步：题目到底要什么

一个 m x n 的网格，机器人站在左上角， 每一步只能往下或往右走，目标是走到右下角。 问：一共有多少条不同的走法？

m, n3, 2

3 行 2 列的网格，从左上角到右下角一共要走 2 步「向下」 + 1 步「向右」， 三步的不同排列方式正好有 3 种 → 答案是 3。

🐢 第二步：暴力搜索为什么会爆炸

最直接的想法：从起点开始，每一步都试「往下」和「往右」两种选择，递归地把所有路径都走一遍， 走到右下角就算一条路径。这样确实能数出答案，但同一个格子会被反反复复地重复经过—— 路径数本身就是指数级的，m、n 稍微大一点（题目里能到 100）就完全跑不动。

关键洞察：机器人走到格子 (i, j) 的「最后一步」只有两种可能—— 要么是从上面 (i-1, j) 往下走一步， 要么是从左边 (i, j-1) 往右走一步，这两种来源完全不重叠。 所以「到 (i, j) 的走法数」就等于「到上面的走法数」加上「到左边的走法数」—— 这是一个典型的动态规划结构：大问题的答案 = 子问题答案的组合。

⚡ 第三步：用二维 DP 表逐格填出答案

设 dp[i][j] 表示「从左上角走到格子 (i, j) 的不同路径数」，于是有：

  • 第一行（i = 0）和第一列（j = 0）是边界： 只能沿着同一个方向一直走，所以这些格子全部是 1。
  • 其余格子：dp[i][j] = dp[i-1][j] + dp[i][j-1]（上面 + 左边）。

按从上到下、从左到右的顺序把整张表填满，右下角的值就是最终答案。 每个格子只需要看一眼它正上方和正左方，时间复杂度从指数级降到了 O(m·n)。

当前正在计算的格子 已经算好的格子 本步依赖的来源格（上 / 左） 终点（最终答案）

试试把 m、n 改成示例里的 3, 3（答案 6）或 7, 3（答案 28）， 看看表格是怎么一格一格被「上面 + 左边」累加出来的。

☕ 第四步：写成 Java 代码

class Solution {
    public int uniquePaths(int m, int n) {
        // dp[i][j]：从左上角走到 (i, j) 的不同路径数
        int[][] dp = new int[m][n];

        // 第一列：只能一直往下走，每个格子都只有 1 种走法
        for (int i = 0; i < m; i++) dp[i][0] = 1;
        // 第一行：只能一直往右走，每个格子都只有 1 种走法
        for (int j = 0; j < n; j++) dp[0][j] = 1;

        // 其余格子：要么从上面下来，要么从左边过来
        for (int i = 1; i < m; i++) {
            for (int j = 1; j < n; j++) {
                dp[i][j] = dp[i - 1][j] + dp[i][j - 1];
            }
        }
        return dp[m - 1][n - 1];
    }
}
时间复杂度 O(m·n)

每个格子只算一次，每次只看上面和左边两个值。

空间复杂度 O(m·n)

用了一张完整的二维表；如果只保留「上一行」，可以压缩到 O(n)。

🚀 小结

套路：当「到达某个状态的方式」可以拆成「来自几个互不重叠的子状态」时， 就把这个状态的答案设成 dp[i][j]，再把它表达成子状态答案的和—— 这正是网格类动态规划的核心。后面遇到「最小路径和」「不同路径 II（带障碍物）」这类题， 都是在这个 dp[i][j] = dp[i-1][j] + dp[i][j-1] 的骨架上做小改动。
