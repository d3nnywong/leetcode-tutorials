贪心算法最经典的入门题之一。难点不在写代码，而在想通一件事： 为什么只记一个数——「目前为止能跳到的最远下标」， 就足以判断能不能到达终点，完全不用去枚举每一种跳法。

🧩 第一步：题目到底要什么

你站在数组的下标 0 处。nums[i] 表示你站在下标 i 时，最多能往右跳几步（也可以跳得比这个短，或者干脆不跳）。 问你能不能跳到最后一个下标。

nums[2, 3, 1, 1, 4]

站在下标 0（值 2）先跳 1 步到下标 1（值 3），再从下标 1 跳 3 步直接跳到下标 4（终点）→ true。

🐢 第二步：笨办法与一个关键发现

最直接的想法：在每个能到达的下标，把所有可能的跳跃长度都试一遍（跳 1 步、跳 2 步……）， 递归地看能不能走到终点。这是指数级的回溯，数组一长就跑不动。

关键发现：我们根本不关心「具体怎么跳」，只关心一件事—— 从已经走过的所有位置出发，最远能跳到哪个下标。 只要这个「最远可达下标」始终能跟上我们当前走到的位置，就一定能继续往前； 一旦它覆盖了终点，答案就是 true；如果走到某个下标， 它已经超过了「最远可达下标」，那不管之前怎么跳，都到不了这里 → false。

⚡ 第三步：贪心，边走边刷新「最远可达」

从左到右扫一遍数组，全程只维护一个变量 farthest：

  • 走到下标 i 时，先检查 i 有没有超过 farthest——超过了就卡住了，直接返回 false。
  • 否则用 i + nums[i] 去刷新 farthest = max(farthest, i + nums[i])。
  • 只要 farthest 已经达到或超过最后一个下标，立刻可以确定能到达，提前结束。

整个数组只扫一遍，O(n)，比指数级的回溯快得多。

当前正在检查的下标 i 已确认能到达的区间 本步刷新出的「最远可达」 卡住：超出了可达范围 终点（已被覆盖）

试试把数组改成 3,2,1,0,4，看看「最远可达」怎么被下标 3 的 0 卡死、再也追不上下标 4。

☕ 第四步：写成 Java 代码

class Solution {
    public boolean canJump(int[] nums) {
        int n = nums.length;
        int farthest = 0; // 目前为止能到达的最远下标
        for (int i = 0; i < n; i++) {
            if (i > farthest) {
                return false; // 前面所有跳跃加起来也够不到这里
            }
            farthest = Math.max(farthest, i + nums[i]);
            if (farthest >= n - 1) {
                return true; // 已经能覆盖终点，提前结束
            }
        }
        return true;
    }
}
时间复杂度 O(n)

数组只扫一遍，每一步只做常数次比较和更新。

空间复杂度 O(1)

只用了 farthest 这一个变量。

🚀 小结

套路：当题目问「能不能到达 / 覆盖某个目标」、而具体路径不重要时， 想想能不能把状态压缩成一个单调不减的边界值（这里是「最远可达下标」）， 边扫边刷新它，把指数级的路径枚举压成一遍线性扫描。这是贪心算法最常见的设计思路之一。
