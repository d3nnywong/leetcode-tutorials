给一堆单词，把「字母重新排一排就能变成同一个词」的那些放进一组。 看完你会明白：给每个单词算一个排好序的「签名」，再用哈希表一遍分组就够了。

🧩 第一步：题目到底要什么

给你一个字符串数组，把字母异位词分到同一组里。 「字母异位词」听着拗口，其实就是：把一个单词的字母重新排列，能不能拼出另一个单词。

strs["eat","tea","tan","ate","nat","bat"]

"eat"、"tea"、"ate" 三个词用的都是 a、e、t 这三个字母，只是顺序不同，互为异位词，分一组； "tan"、"nat" 用的是 a、n、t，分另一组；"bat" 谁也凑不出来，自己一组 → [["bat"],["nat","tan"],["ate","eat","tea"]]（顺序任意）。

🐢 第二步：暴力两两比较为什么慢

最直接的想法：把每个字符串和已经分好的每一组「代表词」两两比较，看是不是异位词 （排序后相不相等，或者数一遍字母出现次数）。这样每来一个新词，都要跟前面所有组挨个比一遍。

慢在哪：最坏情况下是 O(n²·k)（n 个词、每次比较 O(k)）。 问题在于——我们对「这个词属于哪一组」这件事，每次都重新挨个比对， 而不是直接「查」出来。

关键洞察：两个字符串互为异位词，当且仅当把它们的字母都排好序后， 会变成完全相同的字符串。比如 "eat" 和 "tea" 排序后都是 "aet"。 这个排好序的字符串，就是这一组异位词共同的「签名」（指纹）。

⚡ 第三步：用哈希表「按签名分组」

于是思路变成：扫一遍数组，给每个字符串算出它的签名（字母排序后的结果）， 拿这个签名去哈希表里查——有这个签名就把当前词加进对应的组；没有就新建一组。 算签名是查找/比较时唯一要做的事，哈希表让「找到对应组」从 O(组数) 变成 O(1)。

当前字符串 已处理过的字符串 本步新建的组 本步加入的已有组

试试把数组改成 abc,bca,cab,xyz,zzz，看看哪些字符串会被分到同一组。

☕ 第四步：写成 Java 代码

class Solution {
    public List<List<String>> groupAnagrams(String[] strs) {
        // key: 排序后的签名 -> 这一组的所有原字符串
        Map<String, List<String>> groups = new HashMap<>();
        for (String s : strs) {
            char[] chars = s.toCharArray();
            Arrays.sort(chars); // 把字母排好序，得到签名
            String key = new String(chars);
            // 签名相同的字符串，一定会被放进同一个 List
            groups.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
        }
        return new ArrayList<>(groups.values());
    }
}
时间复杂度 O(n · k log k)

n 个字符串，每个最长 k；每个串排序要 O(k log k)，整体扫一遍。

空间复杂度 O(n · k)

哈希表要存下所有字符串本身（以及它们的签名）。

🚀 小结

套路：当「判断两个东西是不是同一类」可以归约成「它们的某种规范化表示是否相等」时， 就把这种规范化表示当作哈希表的 key，扫一遍数组分组，把两两比较的 O(n²) 降到 O(n)。 这道题用「排序后的字符串」做签名；如果字符串只含小写字母，还能用「26 个字母各出现几次」 做签名，把排序的 O(k log k) 降到 O(k)。
