一道「完全背包」的经典变体：每种硬币能用无限次，求凑出目标金额所需的最少硬币数。 看完你会明白：为什么贪心会踩坑、自底向上的一维 DP 又是怎么把它稳稳解决的。

🧩 第一步：题目到底要什么

给你几种硬币的面额 coins（每种硬币可以用无限次）， 以及一个目标金额 amount。问：最少用几枚硬币能正好凑出 amount？凑不出就返回 -1。

🐢 第二步：贪心会踩坑，暴力又太慢

第一反应常常是「贪心」：每次都拿能用的最大面额。但贪心在这题上是 错的——

反例：coins = [1, 3, 4]，amount = 6。 贪心会先拿 4，剩 2，再拿 1、1 → 一共 3 枚（4+1+1）。 但其实 3 + 3 = 6 只需要 2 枚。贪心只看眼前最大， 不会回头比较，所以会错过更优解。

那就只能枚举所有组合：每一步都试遍每种硬币，递归地问「凑出剩下的金额最少要几枚」。 这样能保证对，但同一个「剩下的金额」会被反复递归很多次—— 比如凑 11 可能先减 5 再减 2 剩 4，也可能先减 2 再减 5 剩 4，两条路都会重新算一遍 「凑出 4 最少要几枚」。子问题大量重叠，暴力递归会指数级爆炸。

⚡ 第三步：自底向上，把重复子问题只算一次

既然「凑出金额 j 最少要几枚硬币」会被反复问到，那就把答案记下来， 而且按金额从小到大依次算好——这样轮到大金额时，它需要的小金额答案早就现成了。

定义 dp[j] = 凑出金额 j 所需的最少硬币数（凑不出记为 ∞）。

  • 边界：dp[0] = 0（凑 0 元不需要任何硬币）。
  • 转移：对每个面额 c ≤ j，如果用一枚 c， 剩下的 j − c 已经算好了，那么 dp[j] = min(dp[j], dp[j − c] + 1)。

因为 j − c < j，只要按 j 从小到大算，算到 j 时所有 dp[j − c] 都已经是确定值——这就是「自底向上」。最终答案就是 dp[amount]，如果还是 ∞ 就返回 -1。

当前正在计算的 dp[j] 被参考的来源格 dp[j−c] 本步选中的最优转移 最终答案 dp[amount]

试试把面额改成 1,3,4、amount 改成 6，亲眼看看为什么 dp[6] 会选 3 + 3 而不是贪心的 4 + 1 + 1。

☕ 第四步：写成 Java 代码

class Solution {
    public int coinChange(int[] coins, int amount) {
        // dp[j]：凑出金额 j 所需的最少硬币数；先全部设成「凑不出」
        int[] dp = new int[amount + 1];
        Arrays.fill(dp, Integer.MAX_VALUE);
        dp[0] = 0; // 边界：凑 0 元不需要任何硬币

        for (int j = 1; j <= amount; j++) {
            for (int c : coins) {
                if (c <= j && dp[j - c] != Integer.MAX_VALUE) {
                    dp[j] = Math.min(dp[j], dp[j - c] + 1);
                }
            }
        }
        return dp[amount] == Integer.MAX_VALUE ? -1 : dp[amount];
    }
}
时间复杂度 O(amount · n)

外层金额从 1 到 amount，内层遍历 n 种面额。

空间复杂度 O(amount)

只需要一个长度为 amount + 1 的一维 dp 数组。

🚀 小结

套路：这是「完全背包」的最少件数变体——每种物品（硬币）可以重复选， 求凑出目标值的最少数量。核心都是同一句话：把子问题的答案存下来，按规模从小到大递推， 就能把指数级的暴力递归压成一遍多项式时间的扫描。遇到「贪心直觉不靠谱、子问题又有重叠」的题， 先想想能不能这样定义 dp 数组、写出转移方程。
