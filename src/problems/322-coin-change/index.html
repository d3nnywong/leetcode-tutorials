这道题的规则很简单：给你几种硬币，每种都可以反复使用、用多少次都行， 问凑出目标金额最少要几枚硬币。这类「东西能无限次重复挑」的问题，有个专门的名字叫 「完全背包」——先别管这个名字，不用急着记，读到后面你会发现它其实很自然。 看完你会明白：为什么凭直觉的「贪心」会踩坑、动态规划又是怎么把它稳稳解决的。

🧩 第一步：题目到底要什么

给你几种硬币的面额 coins（每种硬币可以用无限次）， 以及一个目标金额 amount。问：最少用几枚硬币能正好凑出 amount？凑不出就返回 -1。

11 = 5 + 5 + 1，用了 3 枚硬币，是最少的方案 → 返回 3。

🐢 第二步：贪心会踩坑，暴力又太慢

第一反应常常是「贪心」：每次都拿能用的最大面额。但贪心在这题上是 错的——

反例：coins = [1, 3, 4]，amount = 6。 贪心会先拿 4，剩 2，再拿 1、1 → 一共 3 枚（4+1+1）。 但其实 3 + 3 = 6 只需要 2 枚。贪心只看眼前最大， 不会回头比较，所以会错过更优解。

那就只能枚举所有组合：每一步都试遍每种硬币，减掉它，再去问「凑出剩下的金额最少要几枚」。 这一步一步「先减一枚硬币，再把剩下的钱当成一个新的、更小的同类问题去问」的写法， 在编程里有个专门的名字叫递归——就是一个方法自己调用自己去解决一个 更小的同款问题，像照镜子时镜子里还套着一面更小的镜子，或者俄罗斯套娃里面还藏着更小的套娃， 一直套到最小的那个（凑 0 元）才停下来往回收。

这样递归下去确实能保证答案对，但同一个「剩下的金额」会被反复问很多遍—— 比如凑 11 可能先减 5 再减 2 剩 4，也可能先减 2 再减 5 剩 4，两条路都会重新算一遍 「凑出 4 最少要几枚」。这种被反复问到的小问题叫子问题， 它们大量重叠、被一遍遍地白算，暴力递归的计算量会像滚雪球一样越滚越大（指数级增长）， 金额一大就会慢到算不出来。

⚡ 第三步：自底向上，把重复子问题只算一次

既然「凑出金额 j 最少要几枚硬币」会被反复问到，那就把答案记下来， 而且按金额从小到大依次算好——这样轮到大金额时，它需要的小金额答案早就现成了， 不用再重新递归一遍。这种「把子问题的答案存进一张表，每个子问题只算一次」的技巧， 有个正式的名字叫动态规划（英文 Dynamic Programming，很多题解里直接 简写成 DP）。

我们建一张表，用 dp[j] 表示「凑出金额 j 所需的最少硬币数」（这里的 dp 就是 Dynamic Programming 的缩写， 你可以把它当成这张「记事本」的名字；凑不出就记成 ∞，表示「暂时无解」）。 接下来只要想清楚两件事：表格最开头（最简单、不用再往前推的那一格）填什么， 以及每一格怎么由前面已经填好的格子推出来。这两件事在动态规划里分别叫 边界（起点、地基）和转移（用小问题的答案拼出大问题的答案）。

  • 边界：dp[0] = 0（凑 0 元不需要任何硬币，这是最简单、不用再往前推的起点）。
  • 转移：要凑出金额 j，最后一定是用了某一枚硬币 c （前提是 c ≤ j，硬币面值不能比要凑的钱还大）。用掉这一枚之后， 剩下要凑的钱就是 j − c，而这部分最少要几枚早就在 dp[j − c] 里算好了，所以只要在它基础上加这一枚： dp[j] = min(dp[j], dp[j − c] + 1)。 对每种面额 c 都试一遍，取这些结果里最小的那个。

因为 j − c < j，只要按 j 从小到大算，算到 j 时所有 dp[j − c] 都已经是确定值——这就是「自底向上」。最终答案就是 dp[amount]，如果还是 ∞ 就返回 -1。

当前正在计算的 dp[j] 被参考的来源格 dp[j−c] 本步选中的最优转移 最终答案 dp[amount]

☕ 第四步：写成 Java 代码

class Solution {
    public int coinChange(int[] coins, int amount) {
        // dp[j]：凑出金额 j 所需的最少硬币数；先全部设成「凑不出」
        int[] dp = new int[amount + 1];
        Arrays.fill(dp, Integer.MAX_VALUE);
        dp[0] = 0; // 边界：凑 0 元不需要任何硬币

        for (int j = 1; j <= amount; j++) {
            for (int c : coins) {
                if (c <= j && dp[j - c] != Integer.MAX_VALUE) {
                    dp[j] = Math.min(dp[j], dp[j - c] + 1);
                }
            }
        }
        return dp[amount] == Integer.MAX_VALUE ? -1 : dp[amount];
    }
}
时间复杂度 O(amount · n)

外层金额从 1 到 amount，内层遍历 n 种面额。

空间复杂度 O(amount)

只需要一个长度为 amount + 1 的一维 dp 数组。

🚀 小结

套路：这就是前面提到的「完全背包」——每种物品（这里是硬币）都可以重复选， 求凑出目标值的最少数量。核心都是同一句话：把子问题的答案存下来，按规模从小到大递推， 就能把会像滚雪球一样越滚越大的暴力递归，压缩成一遍从头扫到尾、稳稳当当就能算完的动态规划。 遇到「贪心直觉不靠谱、子问题又有重叠」的题，先想想能不能像这道题一样，定义好 dp 数组 （每一格代表什么）、写出转移方程（怎么由小格子推出大格子）。
