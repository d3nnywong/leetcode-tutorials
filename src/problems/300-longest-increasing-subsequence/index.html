这是动态规划里最经典的入门题之一，也是从「逐对比较」走向「贪心 + 二分查找」的代表题。 看完你会明白：先用 O(n²) 的笨办法把思路想通，再看怎么用一个排好序的「影子数组」 把复杂度压到 O(n log n)。

🧩 第一步：题目到底要什么

给你一个整数数组 nums，要你找出最长的一段「严格递增子序列」， 只要返回它的长度就行（不用还原出具体是哪几个数）。

注意「子序列」和「子数组」不一样：子序列不要求连续， 只要求在原数组里删掉一些元素后，剩下的元素相对顺序不变就行。

nums[10, 9, 2, 5, 3, 7, 101, 18]

挑出 2, 3, 7, 101（下标 2、4、5、6），它们在原数组里依次出现且一个比一个大， 长度为 4，这就是最长的一种 → 答案 4。这几个数在原数组里并不挨着， 中间还夹着 9、5、18 等数，这正是「子序列」允许的。

🐢 第二步：先想一个能做出来的办法——动态规划

这里要用的方法叫动态规划（英文缩写 DP）。它的思路说穿了很朴素： 把一个大问题拆成一个个更小的子问题，把每个子问题的答案记下来 （通常存进一个数组里），以后要用到的时候直接翻出来查，不用重新算一遍。

最直接的想法：对每个位置 i，定义 dp[i] 表示 「以 nums[i] 结尾」的最长递增子序列长度——这就是我们准备拿来记账的那张「表」。

要算 dp[i]，就去看它前面所有位置 j < i： 如果 nums[j] < nums[i]，说明 nums[i] 可以接在那段以 nums[j] 结尾的子序列后面，让长度变成 dp[j] + 1。 把所有满足条件的 j 都试一遍，取最大值：

dp[i] = max(dp[j] + 1)，其中 0 ≤ j < i 且 nums[j] < nums[i]

如果没有任何 j 满足条件，dp[i] 就是 1（自己单独成一段）。 算完所有 dp[i]，答案就是它们里的最大值。

慢在哪：外层 i、内层 j 各扫一遍，是 O(n²)。 题目里 n 能到 2500，约六百万次比较还能接受，但官方明确「进阶」要求降到 O(n log n)——问题就出在每算一个 dp[i] 都要把前面重新扫一遍。

⚡ 第三步：贪心维护一个「最小结尾」数组 + 二分查找

换个角度：维护一个数组 tails，tails[k] 表示 「长度为 k+1 的递增子序列」里，结尾数字能取到的最小值。 结尾越小，以后就越容易接上更多的数。

这种「不管以后会不会更好，先让当下这一步占到最划算的位置」的策略，就叫 贪心：每一步都挑眼前看起来最有利的选择，不回头重新权衡——这正是 tails 的核心想法。

可以证明 tails 本身始终是严格递增的（也就是说它自己也是排好序的）， 这正是它能用二分查找的前提。二分查找是一种在「已经排好序」的数据里快速定位目标的办法： 每次只看中间那个数，跟目标比一比，就能立刻排除掉一半的范围——就像玩猜数字游戏， 对方告诉你「大了」还是「小了」，你每次都能把剩下的可能范围砍掉一半，比从头一个一个试快得多。 从左到右扫一遍 nums，对每个数 x：

  • 在 tails 里二分查找第一个 ≥ x 的位置 pos；
  • 如果 pos 等于 tails 的长度（x 比所有结尾都大）： 把 x 接在末尾，tails 变长 → 当前最长长度 +1；
  • 否则：用更小的 x 把 tails[pos] 原地替换掉，长度不变， 但给「以后接上更多数」留了更大的空间。

全程 tails.length 就是「当前看到的最长递增子序列长度」；扫完整个数组即为答案。 二分查找把「找位置」从 O(n) 的线性扫描换成了 O(log n)，这就是 O(n log n) 的关键。

提醒：tails 数组本身不是一条真实存在的递增子序列 （它的元素可能来自原数组里互不相邻、甚至顺序颠倒的位置），它只是用来「记录长度」的辅助结构。 下面的动画会在扫描结束后，额外用 prev 指针回溯还原出一条真正合法的最长递增子序列， 在原数组里用金色标出。
当前 nums[i] / 二分查找的 mid tails（派生出的「最小结尾」数组） mid 可能是答案 / 新接到末尾 mid 太小，需要往右收紧 原地替换 / 还原出的最长递增子序列

试试把数组改成 0,1,0,3,2,3，看看 tails 是怎么一边变长、一边被更小的数替换的。

☕ 第四步：写成 Java 代码

class Solution {
    public int lengthOfLIS(int[] nums) {
        int[] tails = new int[nums.length]; // tails[k]：长度 k+1 的递增子序列的最小结尾
        int len = 0; // tails 当前的有效长度，也就是已知最长递增子序列长度

        for (int x : nums) {
            // 在 tails[0, len) 里二分查找第一个 >= x 的位置（lower bound）
            int lo = 0, hi = len;
            while (lo < hi) {
                int mid = (lo + hi) >>> 1;
                if (tails[mid] < x) {
                    lo = mid + 1; // mid 太小，答案在右边
                } else {
                    hi = mid; // mid 可能就是答案，收紧右边界
                }
            }

            tails[lo] = x; // 新增到末尾，或原地替换成更小的结尾
            if (lo == len) {
                len++; // 比所有结尾都大，长度真的变长了
            }
        }
        return len;
    }
}
时间复杂度 O(n log n)

外层扫一遍数组是 O(n)，内层二分查找是 O(log n)。

空间复杂度 O(n)

额外用了一个长度不超过 n 的 tails 数组。

🚀 小结

套路：当动态规划的「逐一比较」太慢时，可以想想能不能找到一个 单调（有序）的辅助结构，把「线性查找」换成「二分查找」。 这道题里 tails 数组的贪心思想（也叫「耐心排序」patience sorting） 在很多「最长 XX 子序列 / 区间调度」类问题里都会反复出现，值得记住。
