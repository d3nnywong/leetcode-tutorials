一根链表绕成了圈，怎么在不做标记、不开额外空间的前提下发现它？ 答案是让两个指针一快一慢地跑——如果赛道是个圈，快的迟早会从背后追上慢的。

🧩 第一步：题目到底要什么

给你一条单链表的头节点 head，链表里的每个节点只知道 next 指向谁。判断这条链表顺着 next 走下去， 会不会走进一个环——也就是某个节点的 next 绕回了前面已经走过的节点，导致永远走不到尽头（null）。

head[3, 2, 0, -4]
pos1

最后一个节点 -4 的 next 又指回了下标 1（值 2）的节点， 于是 3 → 2 → 0 → -4 → 2 → 0 → -4 → … 永远走不完 → 有环，返回 true。
注意：pos 只是评测系统内部用来「造环」的标记，并不会真的 作为参数传给你的函数——你拿到手的只有一个 head，要自己想办法判断。

🐢 第二步：最直接的笨办法——用哈希表记「走过的节点」

既然不知道有没有环，那就一边走一边把访问过的节点（注意是节点本身，不是值） 存进一个哈希表（HashSet）。每走到一个新节点，先查它在不在表里：

  • 在表里 → 说明这个节点被第二次访问到了，链表有环。
  • 不在表里 → 存进去，继续往后走。
  • 一路走到 null 还没撞见重复节点 → 没有环。
能用，但不是最优：时间 O(n) 没问题，但额外用了一个能装下 所有节点的哈希表，空间 O(n)。题目的进阶要求是 O(1) 空间——一个额外的哈希表都不能要。

⚡ 第三步：快慢指针（Floyd 判圈），O(1) 空间

换个思路：不记录任何节点，只用两个指针。 慢指针 slow 每次走 1 步，快指针 fast 每次走 2 步， 两人从 head 一起出发。

  • 如果没有环：链表迟早会走到尽头，fast 跑得快， 会先于 slow 撞上 null，于是循环结束，返回 false。
  • 如果有环：一旦两人都进了环，就像在环形跑道上赛跑—— fast 每一轮都比 slow 多跑 1 步，差距不断缩小， 迟早会从后面追上 slow，也就是 slow === fast（同一个节点）。这一定会发生，最多再绕一圈。

全程只用了两个指针变量，没有额外的数据结构，空间 O(1)； 指针总共移动不超过 O(n) 步，时间也是 O(n)。

慢指针 slow（走 1 步） 快指针 fast（走 2 步） 相遇节点 / 成环的那条 next

默认演示的就是官方示例 1：[3,2,0,-4]、pos = 1。 也可以把 pos 改成 -1，看 fast 怎么先一步走到 null、判定没有环。

☕ 第四步：写成 Java 代码

// ListNode 由题目给定：
// class ListNode {
//     int val;
//     ListNode next;
// }
public class Solution {
    public boolean hasCycle(ListNode head) {
        ListNode slow = head;
        ListNode fast = head;
        // fast 走两步，必须保证 fast 和 fast.next 都不是 null
        while (fast != null && fast.next != null) {
            slow = slow.next;        // 慢指针走 1 步
            fast = fast.next.next;   // 快指针走 2 步
            if (slow == fast) {
                return true;  // 两者相遇 → 有环
            }
        }
        return false; // fast 走到了 null → 没有环
    }
}
时间复杂度 O(n)

有环时 slow 走不超过一圈、fast 再多绕一圈就能追上；没环时 fast 最多走 n 步到 null。

空间复杂度 O(1)

只用了 slow、fast 两个指针变量，不需要额外的哈希表。

🚀 小结

套路：遇到「链表 / 序列里判断是否会无限循环」的题，先想哈希表记访问过的节点 （直观但要 O(n) 空间），再想能不能用快慢指针把空间压到 O(1)—— 只要存在「一快一慢、差距恒定缩小」的关系，相遇就一定会发生。这个套路后面在 找环入口（142 题）、链表找中点等问题里还会反复用到。
