二叉搜索树有一个「免费」的性质：把它中序遍历一遍，节点的值天然就是从小到大排好的。 这道题教你怎么利用这个性质，不用排序、甚至不用走完整棵树，就能直接拿到第 k 小的答案。

🧩 第一步：题目到底要什么

先把几个词说清楚，免得后面卡壳。树是一种像家谱、像公司组织架构图一样的结构： 一个「节点」下面可以挂着几个「孩子」节点，最上面那个没有「父母」的节点叫根节点 （root）。二叉树就是多加一条规矩：每个节点最多只能有两个孩子， 习惯上叫「左孩子」和「右孩子」。

二叉搜索树（BST，Binary Search Tree）在二叉树的基础上，又多加了一条很有用的规矩： 对树里任何一个节点来说，它左边挂着的所有节点，值都比它小；右边挂着的所有节点， 值都比它大。而且这条规矩不只是根节点自己满足——树上任选一个节点，把它和它下面的孩子单独拎出来看， 照样是「左小右大」。正是这份「天生有序」，才让这道题有花样可玩。

题目给你这样一棵 BST 的根节点 root，和一个整数 k。 要你找出整棵树里第 k 小的节点值 （k 从 1 开始数，k=1 就是最小的那个）。

root[5,3,6,2,4,null,null,1]
k3

把树里所有节点从小到大排一遍：1, 2, 3, 4, 5, 6。第 3 个是 3 → 答案是 3。

🐢 第二步：最直接的笨办法，和它能省掉的部分

最容易想到的办法：把整棵树的节点值都收集到一个数组里，排个序，取下标 k - 1 的那个。这里的「收集」其实就是遍历—— 按某种顺序把树上每一个节点都访问一遍，不漏一个、也不重复访问。能做出来， 但「收集 + 排序」明显是浪费。

浪费在哪？因为二叉树有一种特别的访问顺序，叫中序遍历： 对每个节点，先钻进它的左子树逛一圈，再访问它自己，最后钻进右子树逛一圈 （「中」说的是「自己」排在左、右中间被访问，不是随便起的名字）。而 BST 恰好 满足「左小右大」，所以只要按「左 → 自己 → 右」的顺序走一遍，蹦出来的节点值 序列天生就是从小到大排好的——就拿上面的例子来说， [5,3,6,2,4,null,null,1] 按中序遍历走一圈，正好就是 1, 2, 3, 4, 5, 6，和排序的结果一模一样。根本不需要再排序一次。

关键洞察：既然中序遍历自带顺序，那就一边遍历一边数「这是第几个」， 数到第 k 个立刻停下来，不用等遍历完整棵树。 如果 k 比较小、树比较大，这能省下大量不必要的访问。

⚡ 第三步：用一个显式栈做「能提前停的中序遍历」

中序遍历最自然的写法是递归：写一个函数，让它在处理某个节点时， 直接调用「自己」去处理它的左子树，处理完再调用「自己」去处理右子树—— 一层套一层，像俄罗斯套娃，也像两面镜子对照，镜子里还有镜子。递归写起来很省心， 但有个麻烦：我们数到第 k 个就想立刻喊停、直接把答案返回去， 而递归天然是「一路调用到底、再一路返回」的，没法轻易做到「喊停就退出」 （硬要做也行，得额外弄个标志位到处传，比较绕）。

于是换一种写法：不依赖递归，自己拿一个栈来手动模拟这个过程。 栈是什么？想象一摞叠起来的盘子——你只能从最上面放盘子，也只能从最上面拿盘子， 于是最后放进去的盘子，反而最先被拿出来（这叫「后进先出」）。这里我们用栈来记 「已经路过、但还没轮到访问」的节点。

具体做法：拿一个指针 cur——你可以把它想成一个「书签」， 它不是某个节点本身，只是记着「我现在走到了哪个节点」——从根节点出发，先一路往左压栈到底 （每路过一个节点，就把它当成一只盘子压进栈里，同时指针跟着往左走，直到走到最左边、 没有左孩子为止）。压到头了，就把栈顶弹出来——弹出的这个节点，就是「当前还没访问过的 节点里最小的那个」。每弹出一个就把计数 count 加一， count == k 就直接返回它的值；否则把指针转向它的右子树， 继续「往左压栈到底 → 弹出」的循环。

在栈中等待 当前指针 / 正在出栈 已访问（标了第几个） 第 k 个，最终答案

试试把树改成示例 1 的 3,1,4,null,2、k 改成 1， 看它是不是一路往左走到底，第一次弹栈就停。

☕ 第四步：写成 Java 代码

// Definition for a binary tree node.
// public class TreeNode {
//     int val;
//     TreeNode left;
//     TreeNode right;
//     TreeNode() {}
//     TreeNode(int val) { this.val = val; }
//     TreeNode(int val, TreeNode left, TreeNode right) {
//         this.val = val;
//         this.left = left;
//         this.right = right;
//     }
// }
class Solution {
    public int kthSmallest(TreeNode root, int k) {
        Deque<TreeNode> stack = new ArrayDeque<>();
        TreeNode cur = root;
        int count = 0;

        while (cur != null || !stack.isEmpty()) {
            // 一路往左压栈到底：压到头，栈顶就是「还没访问过的里最小的」
            while (cur != null) {
                stack.push(cur);
                cur = cur.left;
            }
            cur = stack.pop();
            count++;
            if (count == k) {
                return cur.val; // 数到第 k 个，直接返回，不用走完整棵树
            }
            cur = cur.right; // 左边已经访问完，转向右子树继续找更大的
        }
        return -1; // 题目保证 1 <= k <= n，正常不会到这里
    }
}

动画里的「栈 + 指针」和这段代码是逐行对应的：压栈对应内层 while， 出栈对应 stack.pop() 那一刻，count == k 命中就是动画里的 金色高亮。

时间复杂度 O(h + k)

h 是树高（第一次往左压到底要走这么深），之后每弹出一个节点算一次，最多弹 k 次； 最坏情况（k 接近 n）退化为 O(n)。

空间复杂度 O(h)

栈里最多同时压着一条「从根到最左节点」的路径，平衡树 O(log n)，最坏 O(n)。

🚀 小结

套路：遇到「BST + 排名/第 k 小」这类题，先想到 中序遍历自带顺序这个性质，再想要不要用显式栈 把递归摊开，从而获得「提前停止」的能力，省掉不必要的遍历。 如果题目变成「频繁修改 + 频繁查第 k 小」（原题的进阶问）， 就需要在每个节点上额外维护「左子树节点数」，让查询变成 O(log n)。
