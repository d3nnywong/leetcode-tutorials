二叉搜索树有一个「免费」的性质：把它中序遍历一遍，节点的值天然就是从小到大排好的。 这道题教你怎么利用这个性质，不用排序、甚至不用走完整棵树，就能直接拿到第 k 小的答案。

🧩 第一步：题目到底要什么

给你一棵二叉搜索树（BST）的根节点 root，和一个整数 k。要你找出整棵树里第 k 小的节点值 （k 从 1 开始数，k=1 就是最小的那个）。

root[5,3,6,2,4,null,null,1]
k3

把树里所有节点从小到大排一遍：1, 2, 3, 4, 5, 6。第 3 个是 3 → 答案是 3。

🐢 第二步：最直接的笨办法，和它能省掉的部分

最容易想到的办法：把整棵树的节点值都收集到一个数组里，排个序，取下标 k - 1 的那个。能做出来，但「收集 + 排序」明显是浪费—— 因为 BST 的中序遍历（先左子树，再自己，再右子树） 产出的顺序天生就是从小到大的，根本不需要再排序一次。

关键洞察：既然中序遍历自带顺序，那就一边遍历一边数「这是第几个」， 数到第 k 个立刻停下来，不用等遍历完整棵树。 如果 k 比较小、树比较大，这能省下大量不必要的访问。

⚡ 第三步：用一个显式栈做「能提前停的中序遍历」

中序遍历最自然的写法是递归，但递归没法「数到第 k 个就提前退出」（除非额外加标志位，比较绕）。 换成显式栈来模拟就很直观：拿一个指针 cur 从根出发， 先一路往左压栈到底（压栈的同时指针往左走）；压到头了就把栈顶弹出来—— 弹出的这个节点，就是「当前还没访问过的节点里最小的那个」。每弹出一个就把计数 count 加一，count == k 就直接返回它的值；否则把指针转向 它的右子树，继续「往左压栈到底 → 弹出」的循环。

在栈中等待 当前指针 / 正在出栈 已访问（标了第几个） 第 k 个，最终答案

试试把树改成示例 1 的 3,1,4,null,2、k 改成 1， 看它是不是一路往左走到底，第一次弹栈就停。

☕ 第四步：写成 Java 代码

// Definition for a binary tree node.
// public class TreeNode {
//     int val;
//     TreeNode left;
//     TreeNode right;
//     TreeNode() {}
//     TreeNode(int val) { this.val = val; }
//     TreeNode(int val, TreeNode left, TreeNode right) {
//         this.val = val;
//         this.left = left;
//         this.right = right;
//     }
// }
class Solution {
    public int kthSmallest(TreeNode root, int k) {
        Deque<TreeNode> stack = new ArrayDeque<>();
        TreeNode cur = root;
        int count = 0;

        while (cur != null || !stack.isEmpty()) {
            // 一路往左压栈到底：压到头，栈顶就是「还没访问过的里最小的」
            while (cur != null) {
                stack.push(cur);
                cur = cur.left;
            }
            cur = stack.pop();
            count++;
            if (count == k) {
                return cur.val; // 数到第 k 个，直接返回，不用走完整棵树
            }
            cur = cur.right; // 左边已经访问完，转向右子树继续找更大的
        }
        return -1; // 题目保证 1 <= k <= n，正常不会到这里
    }
}

动画里的「栈 + 指针」和这段代码是逐行对应的：压栈对应内层 while， 出栈对应 stack.pop() 那一刻，count == k 命中就是动画里的 金色高亮。

时间复杂度 O(h + k)

h 是树高（第一次往左压到底要走这么深），之后每弹出一个节点算一次，最多弹 k 次； 最坏情况（k 接近 n）退化为 O(n)。

空间复杂度 O(h)

栈里最多同时压着一条「从根到最左节点」的路径，平衡树 O(log n)，最坏 O(n)。

🚀 小结

套路：遇到「BST + 排名/第 k 小」这类题，先想到 中序遍历自带顺序这个性质，再想要不要用显式栈 把递归摊开，从而获得「提前停止」的能力，省掉不必要的遍历。 如果题目变成「频繁修改 + 频繁查第 k 小」（原题的进阶问）， 就需要在每个节点上额外维护「左子树节点数」，让查询变成 O(log n)。
