一个人能不能把这些会议都开了？答案藏在「排序 + 看相邻」里—— 这题是区间冲突检测最经典的入门例子。

🧩 第一步：题目到底要什么

给你一堆会议时间 [start, end]，问：一个人能不能全部参加。 一个人没法同时出现在两个地方，所以只要有任意两场会议的时间重叠， 这个人就没法都参加；只要两两都不重叠，就能全部参加。

输入[[0,30],[5,10],[15,20]]

[0,30] 把 0~30 这段时间全占了，跟 [5,10]、[15,20] 都重叠 → 没法都参加，输出 false。

🐢 第二步：笨办法为什么麻烦

如果会议在输入里是乱序的，想知道某场会议跟哪些会议冲突，得拿它跟 所有别的会议逐一比较时间段是否重叠，两两组合一共要比较 O(n²) 次，慢且容易漏判。

关键洞察：先把会议按开始时间从小到大排好序， 排成「会议 A → 会议 B → 会议 C → …」这样一条队伍。排完之后有个很好用的性质： 如果真有两场会议撞车，那一定能在这条队伍里找到紧挨着的一对也撞车—— 不会出现「A 和 C 撞车，但 A 和挨在中间的 B、B 和 C 却都不撞车」这种情况。

为什么？因为排过序后，B 排在 A、C 中间，说明 B 的开始时间比 A 晚、比 C 早。 如果 A 和 C 撞车，意味着 C 开始的时候 A 还没结束；而 B 比 C 开始得还早， B 一开始的时候 A 当然也还没结束——所以 A 和 B 必然也撞车。 换句话说，「隔着别人偷偷撞车」这种情况根本不会发生，冲突一定会先出现在 相邻的一对身上。于是问题就从「两两比较」简化成了 「从左到右扫一遍，只看紧挨着的会议有没有撞车」。

⚡ 第三步：排序 + 相邻比较，一遍搞定

排好序后，从左到右依次比较相邻两场会议：prev（前一场，队伍里排在前面的那场） 和 cur（当前这一场，紧跟在 prev 后面）：

  • 如果 cur 的开始时间 < prev 的结束时间， 说明这两场会议在时间上重叠 → 直接判定 false，可以提前结束扫描。
  • 如果 cur 的开始时间 ≥ prev 的结束时间， 说明不重叠 → 继续往后比较下一对。

扫完所有相邻对都没发现冲突，就说明这个人能参加全部会议，返回 true。

已比较过 / 当前对中较早的会议 当前正在比较的会议 发现冲突的会议 全部不冲突，可以参加全部会议

试试改成 7,10;2,4，看排序后两场会议完全不挨着，全程没有冲突； 再试试 1,4;4,5，看刚好首尾相接（4 ≥ 4）算不算冲突。

☕ 第四步：写成 Java 代码

class Solution {
    public boolean canAttendMeetings(int[][] intervals) {
        // 按开始时间从小到大排序，重叠的会议排序后一定挨在一起
        Arrays.sort(intervals, (a, b) -> a[0] - b[0]);

        for (int i = 1; i < intervals.length; i++) {
            // 后一场的开始时间 < 前一场的结束时间 → 冲突了
            if (intervals[i][0] < intervals[i - 1][1]) {
                return false;
            }
        }
        return true;
    }
}
时间复杂度 O(n log n)

瓶颈在排序；之后从左到右扫一遍相邻比较只要 O(n)。

空间复杂度 O(log n)

排序所需的额外空间（不算输入数组本身，没有额外的数据结构）。

🚀 小结

套路：遇到「一堆区间，判断是否存在重叠」的题，先按起点 排序，把无序的两两比较关系简化成有序的相邻比较， 再用一次线性扫描搞定，一旦发现冲突还能提前退出。这种「排好序后，从左到右扫一遍， 每一步只根据眼前的信息做判断、绝不回头重新调整」的思路，叫贪心—— 是区间类问题里最常用的套路之一。这和合并区间（56 题）是同一个 「排序预处理 + 贪心扫描」套路，只是这里只需要判断「有没有」冲突，不需要真的合并。
