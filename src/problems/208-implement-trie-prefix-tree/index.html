自动补全、拼写检查背后最常见的数据结构。看完你会明白：一棵树，怎么把一堆字符串的 「公共前缀」压缩共享，让插入和查找都变成「沿着字符往下走几步」这么简单的事。

🧩 第一步：题目到底要什么

要你实现一个叫 Trie（前缀树）的结构，支持三个操作： insert(word) 插入一个单词、search(word) 判断某个单词 是不是被完整插入过、startsWith(prefix) 判断有没有任何插入过的单词 以它为前缀。

操作insert("apple")
操作search("app")

"app" 没有被单独插入过（只插入了 "apple"）→ search("app") 应该返回 false；但 startsWith("app") 应该返回 true， 因为存在以 "app" 开头的单词 "apple"。这一组对比就是这题的核心。

🐢 第二步：先想想笨办法，再看为什么要用树

最直接的想法：拿一个 List<String> 把所有插入过的单词存下来。 search 就遍历一遍看有没有完全相等的；startsWith 就遍历一遍看有没有谁是以 prefix 开头的。

慢在哪：每次查询都要把所有单词扫一遍、逐个字符比较， 单词数一多、单词一长就很慢。而且大量单词其实有公共前缀 （比如 "app"、"apple"、"application" 都以 "app" 开头），笨办法每次都在重复比较这部分。

关键洞察：既然很多单词共享前缀，那就把相同的前缀部分只存一份—— 用一棵树，从根出发，每往下走一条边就代表「拼上一个字符」。多个单词如果前几个字符一样， 它们在树上就走的是同一条路径，到了字符不一样的地方才分叉。

⚡ 第三步：Trie 长什么样，三个操作怎么共用一套逻辑

根节点不代表任何字符，是「还没拼任何东西」的起点。每个节点最多有 26 个子节点 （对应 a~z），边上的字符决定走到哪个子节点。节点自己有一个标记 isEnd：表示「走到这个节点，正好拼出了一个被插入过的完整单词」。

三个操作其实是同一套骨架——从根出发，逐个字符往下走，区别只在两点：

  • 走不到怎么办：某一步没有对应的子节点。 insert 会新建一个节点继续走； search / startsWith 会立刻判定 false。
  • 走到终点之后还要不要看 isEnd： search 走完整个单词后，还要检查终点的 isEnd 是不是 true（光走得到不算数，得是个「完整插入过」的单词）； startsWith 走完就直接算成功，不用管 isEnd （只要存在以它为前缀的词就够了）。
当前指针所在节点 本次操作已走过的路径 走不下去（分支不存在） 操作成功（返回 true / 插入完成） ● 黄点 = 这个节点是某个完整单词的结尾（isEnd）

默认演示就是官方示例的那 6 步操作。试着把序列改成 insert cat, insert car, search cat, search ca, startsWith ca， 看 "cat" 和 "car" 怎么共享前缀 "ca"、又在哪里分叉。

☕ 第四步：写成 Java 代码

class Trie {
    // children[i] 对应字符 'a' + i；为 null 表示这条分支还不存在
    private Trie[] children = new Trie[26];
    // 走到这个节点时，是否正好拼出了一个完整插入过的单词
    private boolean isEnd;

    public Trie() {
    }

    public void insert(String word) {
        Trie node = this;
        for (char c : word.toCharArray()) {
            int idx = c - 'a';
            if (node.children[idx] == null) {
                node.children[idx] = new Trie(); // 没有这条分支就新建
            }
            node = node.children[idx]; // 指针下移一层
        }
        node.isEnd = true; // 走到的终点标记为「单词结尾」
    }

    public boolean search(String word) {
        Trie node = find(word);
        // 不仅要走得到，终点还必须是某个单词的结尾
        return node != null && node.isEnd;
    }

    public boolean startsWith(String prefix) {
        // 只要走得到就够了，不用管是不是单词结尾
        return find(prefix) != null;
    }

    // 三个操作共用的「逐字符下树」骨架：走不到就返回 null
    private Trie find(String word) {
        Trie node = this;
        for (char c : word.toCharArray()) {
            int idx = c - 'a';
            if (node.children[idx] == null) {
                return null;
            }
            node = node.children[idx];
        }
        return node;
    }
}

/**
 * Your Trie object will be instantiated and called as such:
 * Trie obj = new Trie();
 * obj.insert(word);
 * boolean param_2 = obj.search(word);
 * boolean param_3 = obj.startsWith(prefix);
 */
时间复杂度 O(L)

L 是单词或前缀的长度。三个操作都只是沿树走 L 步，和树里已经存了多少单词无关。

空间复杂度 O(Σ L · 26)

最坏情况下每个字符都要新建一个节点，每个节点固定带 26 个子指针的数组。

🚀 小结

套路：当一堆字符串有大量重复的前缀，又要频繁做「精确匹配」或 「前缀匹配」时，用 Trie 把公共前缀压缩成共享路径， 把每次查询的代价从「扫描所有字符串」降到「只走查询本身的长度」。 这正是自动补全、敏感词过滤、IP 路由表这些场景背后常用的结构。
